# 2026-06-08 计划：装载率优化——cap=1 货优先「顶层乘客」逼近理论上限 + 合规诊断

> 架构师交付 Codex 的执行计划。决策依据见 `decision.md`「2026-06-08 已决策」。这是对已落地 stackCapacity 模型的**装载策略优化**，不改堆叠约束语义。

## 背景与根因（样本 `cargo-debug-snapshot (12).json`）

- 柜 20'，货全同尺寸 400×500×600。21 种 218 件，现仅放 **109**。
- cap=1（不可堆叠或 maxStackLayers=1）的货约 **150 件**，可堆货（cap≥3）仅 **68 件**；地面 56 格。
- 实测列高：1 层列 **39 个**（地面全是 cap=1 货，焊死整列）、3 层列 22 个、4 层 1 个。重量 2508/28200，非重量约束。
- 根因：现放置 low-x 逐位铺开，cap=1 货在可堆列尚未建高时就**抢占空地面格**，自己却把那列锁死在 1 层。理论上限约 124（68 可堆 + ~23 顶层乘客 + ~33 地面 cap=1）。

## 任务

### A. cap=1 货优先「顶层乘客」放置（核心）

- **意图**：让 cap=1 的货优先落到「已有堆栈的顶面」，只有找不到合适顶面时才占地面；可堆货继续沉底堆高。最终效果：可堆列尽量堆满，cap=1 货尽量贴到这些列顶上，地面少被 cap=1 货白占。
- **边界**：
  - 不改堆叠**约束**语义（`stackCapacity` / `violatesStackChain` / groundOnly 不动），只改**候选选择/打分偏好**。
  - 不改 weight / input 模式的既有语义；本优化作用于 quantity / volume（与现排序一致的自动模式）。
  - 仍服从所有现有合法性校验（支撑率、容量链、边界、重量、重叠）。
- **模块**：`src/lib/packing.ts` 的候选生成 + `placementScore`（约 `:206`/`:459-470` 一带）。倾向把「该候选是否为 cap=1 货落在已有箱顶面」做成打分项：cap=1 货在顶面候选上**显著加权（更优）**、在地面候选上降权；可堆货维持现有 low-x 深度优先。可抽一个小纯函数 `stackPlacementPreference(item, point, placed)` 便于单测。
- **注意**：避免顾此失彼——可堆货仍需优先沉底建列，否则没有顶面给 cap=1 货用。建议顺序直觉：同一步里可堆货优先建列，cap=1 货优先消费已存在的顶面。

### B. 合规诊断：容量受限提示

- **意图**：当存在未放置货、且未放置主要因 cap=1 货过多 / 地面格用尽（而非重量、尺寸超限）时，在**合规与诊断**给出一条提示，说明「装载已接近此货型的物理上限，受限于不可堆叠/低堆叠容量货物数量」，让用户理解装不满的原因。
- **模块**：`src/lib/packing.ts` 的 `buildDiagnostics`（`:407`）新增一条 `{id:'stack-capacity-limit', severity:'info'|'warning', message}`；经 `reviewChecklist.ts` 的 `diagnostic` 源自动进复核清单/合规与诊断，无需额外 UI。
- **边界**：纯提示，不影响放置结果；文案中英双语与现有诊断风格一致。

## 验证标准 / 测试用例

单测（`packing.test.ts` 或新增 `packing.stackfill.test.ts`，用混合容量夹具，编码业务意图）：

1. **顶层乘客优先**：构造「若干可堆货(cap≥3) + 大量 cap=1 货 + 有限地面」夹具，断言放置后 **cap=1 货落在已有堆栈顶面的数量 > 旧策略**；可堆列被堆满（达到高度上限或货用尽）。
2. **装载率提升**：用样本(12) 等价夹具跑 `calculatePacking`，断言 `placed.length` **显著 > 109**（逼近理论上限方向；给一个下限阈值如 ≥120，最终值以实测为准，但必须明显高于 109）。
3. **不抢地面**：断言「地面格被 cap=1 货占据、其上仍有空间却无法利用」的列数明显下降。
4. **约束未破坏**：所有放置仍满足 `violatesStackChain`（cap=1 货顶上无箱、容量链合法、groundOnly 仅地面）。
5. **诊断提示**：cap=1 受限场景下 `diagnostics` 含 `stack-capacity-limit` 项且文案准确；无未放置货的正常场景**不**出现该提示（避免误报）。
6. **回归**：weight / input 模式放置结果不变；现有 `packing.test.ts` / `packing.31pallet.test.ts` / `manualPlacement.test.ts` 全绿。

命令：`npm run lint && npm test && npm run build`；自动装箱涉及，跑 `npm run test:e2e`。
