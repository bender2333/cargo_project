# 计划：装箱主引擎改用「块构建 + EMS 最合身放置 + 树搜索」（根治贯穿竖缝）

来源：2026/07/07 下午 算法调研 + 真实 debug snapshot 根因坐实 + 用户两轮拍板（选「块构建+树搜索」激进档、允许小件填缝）。根因、调研谱系、选型理由见 `decision.md` 同日下午条目。
本计划**取代** `plans/2026-07-07-ems-space-model.md` 的「纯 EMS 单件放置」定位；EMS 几何模块降级为块放置的底座。

## 为什么换引擎（一句话）
逐箱贪心 + 多 SKU 尺寸拼不齐 → 每 SKU 独立成列、交界锯齿 → 层内空洞层层复制成**从底贯穿到顶的竖缝**（真实数据每层稳定缺 13%，地面空格率 12.7%）。单件放置（含 EMS）改不了列错位；**块构建**先把同 SKU 拼成规整块再摆，从机制上断掉这条因果链。

## 已决策（与用户确认，不再讨论）
1. **主引擎 = 块构建（block-building）+ EMS 最大空区 best-fit 放置 + 有限树搜索前瞻**。
2. **EMS 不废弃**：作为空间追踪底座，供块做 best-fit 选位；不再是唯一候选源。
3. **允许小件填块间边料**（局部混堆）：块之间边料用小件插填，卸货分拣代价用户已接受。混堆区必须在分层/明细如实标注。
4. **fixture 差异先冻结基线→逐条 diff→架构师裁决**，不为过测试弱化断言。

## 必须原样保留（契约）
- `canPlace`（`packing.ts:246-252`）几何门控：越界/重叠/支撑率≥0.5/`respectsMaxStackLayers`/`groundOnly` 落地。块内每箱、填缝每箱都要过。
- `PackingResult` 全字段、`physicalLayer`、支撑链 `supportedBy`、`orientations`、标签贯穿。
- 双模式语义：quantity=最多件数、volume=最高利用率。块构建两模式都要支持（差异在块的排序/取舍）。

## 子任务顺序：1 → 2 → 3 → 4 → 5 → 6。每个单独 commit。

## 子任务 1：冻结基线快照（必做第一步，不改算法）
- **意图**：为「先报差异」建基准。
- **范围**：脚本对当前 `calculatePacking` 跑以下输入，导出 JSON（每箱 id/x/y/z/orientation + 汇总 placed/util/包络填充率/地面空格率/贯穿缝体素占比）：
  - `packing.test.ts`/`31pallet`/`stackfill` 全部装箱输入；
  - `test-data/excel/越南第十一批6.2海运.xlsx` × 40HQ、以及 `cargo-debug-snapshot (15).json` 对应输入 × 20GP，各 quantity/volume。
- **验证标准**：基线 JSON 落盘 `test-data/json/baseline-2026-07-07/`。这是后续差异裁决唯一基准。

## 子任务 2：EMS 空间模型（纯几何模块 + 完整单测）
- **意图**：把「空的最大长方体」做成可独立测试的纯函数，作为块放置底座。（沿用上午 ems 计划的模块设计。）
- **范围**：新建 `src/lib/emsSpace.ts`：`initEMS(container)`、`splitEMS(emsList, placedBox)`（切相交 EMS 出子块 + `pruneContained` 剔冗余）、`emsBestFit(emsList, blockSize)`（返回最合身的 EMS 及其近原点角——块放置用 best-fit，不是任意角）。
- **验证标准（可断言单测）**：空柜放1块→剩余 EMS 恰覆盖 L 型且两两不含；放两不相邻块→中间空档作为独立 EMS 存在（核心断言）；`pruneContained` 剔除被含 EMS；`emsBestFit` 对给定块尺寸返回浪费最小的 EMS。纯函数，不 import packing.ts。

## 子任务 3：组块（block generation，纯模块 + 单测）
- **根因**：同 SKU 独立成列、交界锯齿是缝的来源；规整块可对齐。
- **意图**：新建 `src/lib/blocks.ts`。把每个 SKU 的 N 个相同箱组合成若干候选**块**（block = 某朝向下 `nx×ny×nz` 个同箱拼成的大长方体，尺寸=箱尺寸整数倍，块内零缝、支撑100%）。
  - 生成同 SKU 的候选块集合（不同 nx/ny/nz 组合、不同朝向），受柜内尺寸与该 SKU 的 `maxStackLayers`/`groundOnly`/`stackable` 约束（不可堆的 nz=1）。
  - 提供块的度量：体积、底面积、尺寸，供放置阶段排序/best-fit。
- **边界**：纯函数，输入 CargoItem + 容器尺寸，输出候选块列表。不放置、不 import packing 主循环。
- **验证标准（可断言单测）**：
  - 126 个 530×305 箱、柜宽2352 → 能生成含 `ny=7`（7×305=2135≤2352）的块候选，且块宽≤柜宽。
  - `groundOnly` SKU 的所有候选块 `nz=1`。
  - `maxStackLayers=k` 的块 `nz≤k`。
  - 块内零缝断言：块体积 == 箱体积 × nx×ny×nz。

## 子任务 4：主循环重写 — 树搜索选块 + EMS 放置 + 小件填缝（核心）
- **意图**：替换 `packing.ts` 逐箱极值点贪心（`:830-968`）为：
  1. 组块（子任务3）→ 得到所有 SKU 的候选块。
  2. **有限树搜索/前瞻**选「下一个放哪个块、用哪个朝向、放哪个 EMS」——评价函数 = 放置后剩余空间可用性（best-fit 浪费最小 + 利用率增益）。深度/宽度设上限（如 beam width、look-ahead 步数），可退化为纯贪心 best-fit。
  3. 用 `emsBestFit` 落块，`splitEMS` 更新空间；块内每箱走 `placeEntry`，仍过 `canPlace`。
  4. **小件填缝阶段**：主体块放完后，对剩余 EMS 用小件（单箱或小块）best-fit 填充，直到无 EMS 可容纳任何剩余件。填缝箱标记「混堆填充」来源（供子任务5 呈现）。
- **保留**：`placementScore` 的启发式在「块内朝向选择」「填缝选位」仍可复用其防倾倒/标签朝门/贴边逻辑；不必整段删。`canPlace` 是最后防线。
- **边界**：两模式都走此引擎，差异只在块/件的排序权重（quantity 偏多件、volume 偏大块优先）。不改几何判定、不改 PackingResult 结构。
- **验证标准**：
  - 20GP + snapshot 数据：**包络填充率 > 88%**（基线 82.4%）、**地面空格率 < 8%**（基线 12.7%）、贯穿缝体素（每层缺口）显著下降；util 不低于基线。
  - 40HQ + 越南十一批：util 显著 > 76.5%。
  - 无重叠/越界/支撑违规（几何断言，脚本校验全部 placed）。
  - 性能：864 箱两模式各跑，记录耗时；给上界 ≤ 5s，超则调小树搜索宽度/降级贪心并记 decision.md。
  - 子任务1 基线逐 fixture diff → 差异清单交架构师。

## 子任务 5：混堆填缝的呈现（fail-loudly，不静默）
- **意图**：小件填缝会让某区域非单一 SKU。分层/明细/卸货视图须如实标注该区含混合货品，避免卸货误判。
- **范围**：填缝箱在 `PackingResult` 里可识别（来源标记）；分层视图/明细表对混堆区给出提示或分组标注。中英文。
- **验证标准**：构造含填缝的用例，断言填缝箱可被识别且在明细/分层中标注；E2E 目视确认提示存在。

## 子任务 6：回归收口
- **意图**：fixture 差异裁决 + 全量回归。
- **流程**：差异「util升/缝减/无违规」→改善，更新硬坐标断言并注释「块构建重写后坐标 2026-07-07」；「util降/件减/违规」→回归，记 decision.md，回子任务4 修，不改测试。架构师确认后才动断言。
- **验证标准**：`npm run lint && npm test && npm run build` 全绿；`npm run test:e2e` 全绿（失败先记 decision.md）；差异裁决记录入 decision.md。

## 风险与回归门槛
- **最大轮次改动**：主循环全重写。已定基线冻结→逐条 diff→架构师裁决，不弱化断言。
- 组块/EMS 有 bug 会产生重叠——子任务2/3 单测覆盖切分/剔除/块内零缝/约束；`canPlace` 是运行时最后防线（重叠必拒）。
- 树搜索性能：深度/宽度设硬上限，可退化贪心；864 箱实测，超 5s 降级并记 decision.md。
- 混堆填缝须 fail-loudly 呈现，否则卸货端静默出错。
- `loadingPriority` 残留清理**不在本轮**（避免混 churn），维持现状。
- 若块构建 util 未达 88% 下界：先查组块是否覆盖足够块型（尤其边料利用），再评估是否加深树搜索；仍不足记 decision.md，不强行改断言充数。
