# 计划：双模式装箱（数量优先 / 体积优先），去除优先级字段

来源：2026/06/30 与用户多轮讨论后的**方向修订**。用户拍板：件数与容量是冲突目标，**用「双模式」让用户选**，去掉每件货的装载优先级字段，不引入「必装」硬约束。
本计划**取代** `plans/2026-06-29-loading-priority-and-packing-fill.md` 与 `plans/2026-06-30-greedy-defrag-global-bestfit.md` 的算法/优先级部分。根因与实验数据见 `decision.md`。

## 已决策（与用户确认）

1. **双模式 = 复用现有 `loadingMode`**：
   - **数量优先（quantity）** = 小件先装 = **最大化件数**。
   - **体积优先（volume）** = 大件先装 = **最大化利用率 + 整托（最大件）自然全装**。
2. **去除 `loadingPriority` 字段**（回退 Codex 子任务1/4 中该字段相关改动）。理由：优先级字段与双模式是两套表达意图的机制，并存会冲突（CLAUDE.md「暴露冲突别平均」「Simplicity First」）。
3. **不引入「必装」硬约束**。用户靠选模式表达意图：要整托全装就选体积优先，要件数最多就选数量优先。
4. **保留 `groundOnly`（必须落地）**：这与优先级无关，是玻璃 C 不可上托的独立业务约束，仍需要。

## 实验依据（真实 0629 数据，见 decision.md）

| 排序/模式 | 件数 | 利用率 | 整托A | 散货 |
|---|---|---|---|---|
| 小件先（数量优先现状 0.8） | 274 | 77.1% | 2/10 | 全装 |
| 小件先（数量优先现状 0.5） | 273 | 71.7% | 1/10 | 全装 |
| 大件先（体积优先应有行为） | 86 | 78.6% | **10/10** | C71+D4（散货填充弱） |
| 当前 volume（全局最优） | 166 | 80.7% | 7/10 | C103+D55 |

- 件数 vs 利用率是固有冲突：小件先件数多、利用率低；大件先利用率高、整托全装、但散货填充差（75/272）。
- 大件先排序即可让整托全装（无需优先级字段）。
- 散货填充弱是极值点模型不追踪最大空区的遗留问题（R-D），靠托顶/缝隙点缓解。

## 根因（file:line）

- 数量优先掉利用率：`canPlace` 阈值被改 0.5（受控实验证 0.8→0.5 掉 5.4%=少一托A，机制见 decision.md）。
- 体积优先只装7托：当前 volume 全局 best-fit（`packing.ts:830+`）贪体积，装7托后转投散货，剩3托被丢。应改为大件优先保证整托全装。
- C=0 bug：`reserveTopPassengerStackSlot`/`minPendingTopPassengerHeights`（`packing.ts:917-921` 等）与已加的优先级排序交互，散货 C 被全判 no-space。
- 散货填充弱：极值点 `placeEntry` 不追踪最大空区（R-D，本轮靠托顶/缝隙点缓解，不重写空间模型）。

## 子任务 1：移除 loadingPriority 字段 + 清理优先级/预留逻辑

- **意图**：撤掉「装载优先级」这套机制，回到「双模式」单一表达。
- **范围**：
  - `src/types.ts`：移除 `CargoItem`、`PlacedBox`、`ImportTemplateDefaults` 的 `loadingPriority`（回退 Codex `2e40f1f`）。**保留 `groundOnly`**。
  - `packing.ts`：移除 `loadingPriorityRank`、优先级排序分支（`:737,746` 等）、`canUseTopSurfacePoints` 的 `loadingPriority` 条件、`reserveTopPassengerStackSlot`/`minPendingTopPassengerHeights`「顶部乘客预留」逻辑（连带修掉 C=0 bug R-C）。
  - `importCargo.ts` / `ImportMappingForm.tsx` / `Workbench.tsx` / `server`：移除 loadingPriority 的解析/别名/模板字段/后端校验（回退 `977988b` 中该字段部分）。**保留 groundOnly 的导入贯穿**。
- **边界**：不改 groundOnly 相关；不改分层/几何判定。
- **验证标准**：`npm run build` 通过；移除后无 loadingPriority 残留引用（grep 为 0）；现有测试全绿（剔除依赖 loadingPriority 的用例，若 Codex 之前加过）。

## 子任务 2：数量优先模式 — 回退阈值 0.8，恢复件数最大

- **根因**：0.5 阈值导致件数模式掉一托 A / 利用率。
- **意图**：`canPlace` 阈值回退 **0.5 → 0.8**（数量优先场景）；恢复小件先装排序（stackCapacity→数量降序）。
- **边界**：只改阈值常量与排序；不动引擎结构。阈值是否按模式区分（volume 用 0.5）**推迟到子任务4实测后定**，本子任务先用 0.8 修回归。
- **验证标准（0629 数据）**：数量优先模式断言 `placed ≥ 274`、`util ≥ 77.1%`（回到 0.8 基线，修掉 71.7% 回归）。防回归：`packing.test.ts`/`stackfill`/`31pallet` 全绿。

## 子任务 3：体积优先模式 — 大件优先，整托全装

- **根因**：当前 volume 全局 best-fit 只装7托。
- **意图**：体积优先模式改为**大件优先排序**（按体积或底面积降序），保证最大件（整托）在散货之前全部装入。机制择一（实现时定）：
  - 方案A：改为「体积降序 sequential」放置（实测大件先=10托全装）。
  - 方案B：保留全局 best-fit 但强化大件优先权重，使整托先于散货装完。
  - **倾向方案A**：简单、可预测、直接保证整托全装；若利用率明显低于现状 80.7% 再考虑 B。
- **边界**：只改体积优先模式的排序/放置策略；不改几何判定。保留现有 volume 模式测试作为参照，结果变化逐条判定改善 vs 破坏。
- **验证标准（0629）**：体积优先断言**整托 A 全 10 托 + B 装入**；`util ≥ 78%`（大件先基线）。若低于当前 80.7% 需记 decision.md 权衡（整托全装优先于极限利用率，符合业务）。

## 子任务 4：散货填充增强 + 阈值复测

- **根因**：整托铺完后散货只填 75/272（R-D）。
- **意图**：为散货填充引入**托顶网格点 + 缝隙点**候选（`topSurfacePoints` 现成，放开对可堆叠货物的使用），让整托顶面与柜内缝隙尽量填满。两模式都受益，尤其体积优先。填充完成后**用 0629 数据复测 0.8 vs 0.5 阈值**：若 0.5 在填充场景显著提升散货件数且不伤件数模式，则按模式采纳（如体积优先 0.5、数量优先 0.8）；否则统一 0.8。用数据定。
- **边界**：不重写空间模型（层次2 不做）；只扩候选点集 + 阈值实测。
- **验证标准（0629）**：体积优先模式散货件数**显著高于 75**（给下界如 C+D ≥ 120，实测收紧）；整托仍全装；数量优先不退化。阈值决策与实测写入 decision.md。
