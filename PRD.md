# 货柜装箱计算系统重构 PRD

## 1. 项目背景

当前项目已经迁移到 Vite + React + TypeScript 架构，并具备基础装箱计算、3D 显示、标签、Excel 导入导出和中英文切换能力。但当前前端体验与 `archive/` 中旧版交付页面存在差异，且现有“分层查看”不能准确表达真实装柜过程中货物如何从底层开始逐层堆叠。

本次重构目标是：参考 `archive` 旧版工作台的界面和功能组织方式，在当前技术栈下重建产品体验，同时保留并强化现有 3D、标签、导入导出和计算能力。

参考对象：

- `archive/active-8080/index.html`
- `archive/container-layout/index.html`
- 当前 `src/` 中的装箱计算、3D 场景、Excel 导入导出与双语实现

## 2. 产品目标

1. 使用 Vite + React + TypeScript 完成旧版工作台体验的现代化重构。
2. 保留现有 3D 显示、标签、Excel 导入导出、计算逻辑等后台能力。
3. 将标签功能作为核心业务能力贯穿录入、计算、展示、导出和历史方案。
4. 重构分层查看，使其符合真实装柜作业视角，而不是简单按高度过滤。
5. 暂不考虑多用户、权限、在线协作、账号、许可证等管理类功能。

## 3. 用户与使用场景

主要用户是物流、运营、装柜规划和业务审核人员。

典型流程：

1. 选择货柜类型或录入自定义柜体参数。
2. 手动录入或通过 Excel 导入货物。
3. 通过标签识别不同货物类型。
4. 点击装箱计算，生成装柜方案。
5. 查看 2D、3D、明细、分层和诊断结果。
6. 根据层级和标签判断装柜顺序、空间利用率和未装入原因。
7. 导出或打印方案。

## 7. 货物与标签需求

标签是本系统的核心业务功能，不是普通展示字段。

### 7.1 货物字段

每个货物箱型至少包含：

- `label`：短标签，例如 `A`、`B`、`C1`
- `name`：货物名称
- `length`
- `width`
- `height`
- `weight`
- `quantity`
- `color`
- `canRotate`
- `stackable`

### 7.2 标签使用范围

标签必须贯穿以下环节：

- 手动录入
- Excel 导入
- 货物清单
- 装箱计算
- 2D 视图
- 3D 视图
- 分层查看
- 明细统计
- 未装入原因
- Excel 导出
- 历史方案恢复

### 7.3 标签展示要求

- 同一标签的箱体颜色必须一致。
- 2D 和 3D 中必须能看出箱体标签。
- 当货物发生水平旋转、侧放或倾倒时，标签文字方向必须跟随实际朝向变化，不能始终固定正向显示。
- 明细中必须按标签统计计划数量、已装数量、未装数量。
- 导入、导出和恢复历史方案时不得丢失标签。
- 支持按标签高亮或筛选查看货物位置。

## 8. 装箱计算需求

装箱结果必须满足以下硬约束：

- 箱体不得超出货柜有效边界。
- 总重量不得超过最大载重。
- 箱体不得互相重叠。
- 不可旋转货物不得变换朝向。
- 不可堆叠货物上方不得放置其他货物。
- 被堆叠货物必须有明确支撑关系。
- 图上展示的所有箱体必须是合法计算结果。

计算结果至少包含：

```ts
type PackingResult = {
  placed: PlacedBox[]
  unplaced: UnplacedCargo[]
  layers: PackingLayer[]
  workSteps: LoadingStep[]
  labelStats: LabelPackingStats[]
  diagnostics: PackingDiagnostic[]
}
```

每个 `PlacedBox` 必须保留用于展示的朝向信息，避免 2D、3D、明细表各自通过尺寸猜测旋转：

```ts
type PlacedBoxOrientationInfo = {
  orientationKey: 'LWH' | 'WLH' | 'LHW' | 'HLW' | 'WHL' | 'HWL'
  labelRotationDeg: 0 | 90 | 180 | 270
}
```

其中 `orientationKey` 表示原始长宽高映射到实际放置长宽高的顺序，`labelRotationDeg` 表示标签在可视化层需要采用的离散旋转角度。

## 9. 分层查看重构需求

当前分层查看需要重构。新分层不能只按 `z` 高度粗略过滤，而要表达真实装柜过程中货物如何一层层堆叠。

### 9.1 分层概念

系统需要区分三类层级：

1. 物理高度层：根据实际空间位置和支撑关系生成。
2. 装柜作业层：从工人视角表达铺底、堆叠、推进的顺序。
3. 标签分布层：展示某一层中不同标签货物的分布。

### 9.2 箱体层级字段

每个已装箱体需要包含：

```ts
type PlacedBoxLayerInfo = {
  physicalLayer: number
  workStep: number
  supportType: 'floor' | 'fully-supported' | 'partially-supported'
  supportedBy: string[]
  label: string
}
```

### 9.3 分层规则

- 接触地面的箱体为第 1 层。
- 放在其他箱体上的箱体进入更高层。
- 混合高度货物不能只按相近 `z` 值归为同一层。
- 分层必须基于支撑关系、底面接触区域和堆叠关系。
- 层级数据必须与 2D、3D、明细表一致。

### 9.4 分层交互

必须支持：

- 查看全部层
- 查看当前层
- 上一层 / 下一层
- 其他层透明显示
- 当前层高亮
- 当前层标签显示
- 当前层统计
- 当前层标签分布
- 支撑关系提示
- 装柜作业步骤查看

当前层统计包括：

- 当前层箱数
- 当前层重量
- 当前层体积
- 当前层涉及标签
- 各标签数量

## 10. 2D 视图需求

2D 视图需要服务于装柜理解，而不仅是简单俯视图。

必须支持：

- 俯视图
- 正视图
- 侧视图
- 当前层查看
- 标签显示
- 当前层高亮
- 其他层透明
- 按标签高亮
- 导出当前视图

2D 视图需要回答：

- 当前层有哪些标签？
- 每个标签分别在哪里？
- 当前层是否有明显空隙？
- 当前层是否能支撑上一层？
- 装柜顺序是否符合实际操作视角？

## 11. 3D 视图需求

保留并增强当前 3D 能力。

必须支持：

- 箱体按真实比例展示
- 箱体颜色与标签一致
- 箱体表面显示标签
- 标签方向跟随 `PlacedBox.orientationKey` 和 `labelRotationDeg` 显示，用户能通过标签判断箱体是否旋转、侧放或倾倒
- 鼠标旋转、缩放、查看不同角度
- 等轴、俯视、正视、侧视视角
- 分层切换时当前层高亮
- 非当前层透明或弱化
- 浏览器窗口最大化或尺寸变化后，3D 画布必须跟随外层工作区重新布局和重新设置 WebGL 尺寸，不能停留在小视口
- 自由视角下所有已装箱体必须保持可观察，不得因为透明材质排序、背面剔除或相机角度造成局部箱体/面片缺失

验收时不得出现：

- 明细表显示在第 2 层，但 3D 中表现为第 1 层
- 当前层统计数量与 3D 可见数量不一致
- 标签在 3D 中丢失或无法识别
- 浏览器最大化后 3D 区域不扩展
- 自由旋转视角后已装箱体局部消失

## 12. Excel 导入导出需求

### 12.1 导入

支持 `.xlsx`、`.xls`，可兼容 `.csv`。

导入后必须进入统一货物数据源，并参与后续装箱计算、标签显示、分层查看和导出。

导入字段应支持：

- 标签
- 名称
- 长
- 宽
- 高
- 重量
- 数量
- 颜色
- 是否可旋转
- 是否可堆叠

如字段缺失或无法识别，应显示明确错误或警告，不得静默导入失败。

### 12.2 AI 辅助导入预留

系统需要为后续 AI 字段映射预留接口，用于处理真实业务 Excel 中的中文字段、单位差异和非标准表头。

示例映射：

- `托盘` → `label`
- `整托重量kg` → `weight`
- `长cm` → `length`
- `宽cm` → `width`
- `高cm` → `height`

如源字段为厘米，应转换为系统内部统一单位。

### 12.3 导入映射弹窗设计

智能字段映射不能只显示下拉框。映射弹窗需要同时展示“字段、单位、原始数据、转换结果”四类信息，帮助用户确认真实业务 Excel 是否被正确识别。

弹窗至少包含：

- 原始表头列表和系统字段映射表。
- 前 5-10 行数据预览，保留原始单元格值。
- 长、宽、高字段的单位选择，至少支持 `mm`、`cm` 和自动识别。
- 单位选择变化后展示换算后的毫米预览。
- 必填字段缺失、重复映射、不可解析数值的行级提示。
- 确认导入前显示预计导入行数、错误行数、警告行数和厘米换算行数。

交互要求：

- 用户可以修改字段映射。
- 用户可以修改尺寸单位判断。
- 用户确认前不得静默覆盖当前货物数据。
- 弹窗只负责把 Excel 转换成标准 `CargoItem[]`，不得把导入解析逻辑写入 UI 组件。

### 12.4 导出

导出结果应包含：

- 标签
- 名称
- 原始尺寸
- 实际朝向
- 重量
- 计划数量
- 实际排入数量
- 未装入数量
- 层级
- 作业步骤
- 失败原因

## 15. 非功能需求

- 正确性：计算结果可重复，所有可视化结果必须来自合法计算数据。
- 性能：常规业务数据量下，计算与视图切换应保持流畅；3D 自由视角旋转、浏览器窗口 resize、侧栏折叠/展开不得造成明显卡顿或持续资源泄漏。
- 可维护性：UI、计算逻辑、导入导出、3D 渲染应模块化。
- 可测试性：核心算法、分层逻辑和导入逻辑应有自动化测试。
- 可扩展性：保留 AI 导入、更多柜型、更多规则的扩展空间。

## 16. 技术原则

- 使用 Vite + React + TypeScript 作为唯一主技术栈。
- 旧版 `archive` 作为功能和视觉参考，不直接依赖旧版静态 bundle。
- 装箱计算逻辑不得写死在 UI 组件中。
- 分层逻辑应独立成可测试模块。
- 3D 组件应消费标准化后的 `PackingResult` 数据。
- 标签应作为核心数据字段，不应只在渲染层临时生成。

## 17. 验收标准

1. 页面结构与旧版工作台体验一致，且运行在当前 Vite + React + TypeScript 架构中。
2. 多用户、权限、账号、许可证功能不作为本期有效入口。
3. 标签在录入、导入、计算、2D、3D、分层、导出和历史中完整保留。
4. 分层查看基于真实堆叠和支撑关系，而不是简单高度过滤。
5. 2D、3D、明细表中的层级和标签一致。
6. 混合高度货物场景下，层级仍然正确。
7. Excel 导入后的货物能直接参与装箱计算。
8. 3D canvas 正常渲染、可交互、可切换层级。
9. 未装入货物必须有清晰原因。
10. 单元测试和 Playwright E2E 覆盖核心流程。
11. 浏览器窗口最大化后，3D 工作区与 WebGL canvas 尺寸同步扩大。
12. 自由视角旋转过程中不出现已装箱体局部消失。
13. 旋转货物的标签在 2D 和 3D 中体现朝向变化。
14. Excel 智能映射弹窗展示表头、样例数据、字段映射、单位选择和转换预览。
