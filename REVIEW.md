## 第三十轮 Review 与「阶段合并装柜步骤图」计划

日期：2026-06-02

本轮 review 针对用户提出的「类似 EasyCargo 的任务分解图」需求。经讨论确认，这里的产品语义不是项目任务拆解，而是面向现场执行的**装柜步骤图**：把自动装箱结果转成可播放、可打印、可按标签识别的分步装柜说明。

本轮已从 review 计划推进到第一版实现：新增阶段合并业务模块、结果区「装柜步骤」Tab、当前阶段高亮和 Playwright 覆盖。打印/导出步骤图仍按计划留作后续小步任务。

### 1. 问题确认：当前有顺序数据，但没有现场作业级步骤图

当前系统已经有装柜作业顺序数据：

- `PackingResult.workSteps`
- `PlacedBox.workStep`
- `PlacedBox.physicalLayer`
- `PlacedBox.supportedBy`
- 标签、颜色、支撑类型和层级统计

但这些数据目前主要服务回放、分层和明细展示，仍不是 EasyCargo 式的现场作业说明。用户需要在 2D、3D、分层、明细和回放之间来回理解：

- 哪一批箱体应先装。
- 当前阶段包含哪些标签和数量。
- 当前阶段位于货柜哪一段深度、哪一层、哪一侧。
- 当前阶段与下一阶段之间的作业边界是什么。
- 是否能直接打印给装柜现场使用。

因此下一阶段目标不是重写装箱算法，而是在现有 `workStep` 语义之上生成可执行的装柜阶段。

### 2. 决策：第一版采用阶段合并，不采用每箱一步

用户已确认第一版使用**阶段合并**。

不采用每箱一步的原因：

- 真实业务数据可能一次装入上百个箱体；如果每箱一步，步骤图会变成 100 到 200 个页面或动画节点，现场不可用。
- 同标签、同层、相邻深度/宽度区域的箱体通常可以作为一个连续作业阶段执行。
- 当前第二十九轮已把 `workStep` 修正为最终深度优先顺序，适合在此基础上合并为阶段，而不是重新发明顺序。

阶段合并的初版边界：

- 合并来源仍是 `result.workSteps` 对应的 `PlacedBox`，不新造另一套顺序。
- 优先按连续 `workStep` 合并，保证阶段顺序与回放顺序一致。
- 同一阶段应尽量满足：同一 `physicalLayer`、同一标签或少量相邻标签、相邻深度区间、相同或兼容支撑状态。
- 阶段必须保留箱体明细，允许从阶段回钻到每个 `PlacedBox`。
- 如果相邻箱体跨层、跨明显深度段、支撑关系变化明显，必须拆成不同阶段，不能为了减少步骤数损害现场理解。

### 3. 拟新增业务模型

下一阶段优先新增纯业务模块，例如 `src/lib/loadingTaskGroups.ts`，从 `PackingResult` 生成阶段数据：

```ts
type LoadingTaskGroup = {
  id: string
  sequence: number
  stepStart: number
  stepEnd: number
  physicalLayer: number
  labels: Array<{ label: string; color: string; count: number }>
  boxIds: string[]
  bounds: { xMin: number; xMax: number; yMin: number; yMax: number; zMin: number; zMax: number }
  supportTypes: PlacedBox['supportType'][]
  supportedBy: string[]
  summary: string
}
```

这个模块必须单元测试覆盖：

- 连续同层同标签箱体会合并。
- 跨 `physicalLayer` 必须拆分。
- 跨明显深度段必须拆分。
- 支撑状态变化明显时必须拆分。
- 阶段 `stepStart/stepEnd` 与 `boxIds` 不丢失任何已装箱体。

### 4. 已实现：新增「装柜步骤」结果页

结果区新增 Tab：

- 中文：`装柜步骤`
- 英文：`Stage Plan`（避免和主操作按钮 `Load` 产生可访问名称匹配冲突）

第一版已实现：

- `src/components/LoadingStepsPanel.tsx` 展示阶段列表、阶段编号、标签统计、数量、层级、深度/宽度/高度区间和支撑信息。

- 提供上一阶段、下一阶段、播放/暂停。
- 支持按阶段查看箱体明细。
- 保留标签颜色和标签文字，不能退化成无业务含义的色块动画。
- Playwright 覆盖计算后进入「装柜步骤」、阶段切换、标签统计和阶段合并数量。

本轮暂不做：

- 工人派工、多用户协作、任务状态回传。
- 复杂路线规划或搬运路径仿真。

## 第二十九轮 Review 与「深度优先装柜顺序」修复计划

日期：2026-05-30

本轮 review 针对用户提供的复现快照：

- `C:\Users\BA_H3C_Pad\Downloads\cargo-debug-snapshot (4).json`

### 1. 问题确认：顶部补装箱被排到外侧深度之后

快照中自动排布共有 210 个箱体，货物尺寸均为 `400 × 500 × 600`，允许旋转。几何结果本身没有越界或重叠，但作业顺序不符合「从集装箱里边往外装」的深度优先语义。

关键证据：

- `physicalLayer=1` 对应最内侧 `x=0..400`，但它的 `workStep` 分布到了 `1..171`。
- 最内侧顶部补装箱位于 `x=0,z=1800`，对应步骤 `169..171`。
- 第一个外侧深度箱 `x=400` 已在 `workStep=13` 出现。
- 这意味着回放/作业清单会先进入外侧深度，再很晚才回到最内侧顶部，形成用户指出的「最上边最后安装」。

### 2. 根因：摆放顺序与作业顺序混用了同一个早期编号

代码定位：

- `src/lib/packing.ts` 在 `placeEntry()` 中用 `placed.length + 1` 立即写入 `workStep`。
- 随后才调用 `assignDepthLayers(placed)`，把分层语义改为沿货柜深度 `x` 从里到外。
- 但 `workStep` 没有跟随深度层和最终坐标重新编号，仍保留极点贪心生成时的插入顺序。

因此，这不是单个箱体坐标错误，而是「最终装柜作业顺序」没有在深度优先语义确定后重新生成。

### 3. 已形成回归单元测试

新增测试：

- `src/lib/packing.test.ts`：`loads rotatable top-fill boxes before moving to the next outer depth slice`

测试覆盖与快照等价的场景：

- 20GP 柜型。
- 多批 `400 × 500 × 600` 可旋转货物。
- 断言最内侧 `x=0,z>=1800` 的顶部补装步骤必须早于第一个 `x>0` 外侧深度步骤。

这个测试在修复前失败：

- 失败值：`expected 171 to be less than 13`

### 4. 修复方向

本轮采用最小修复，不改变几何摆放算法：

- 保留极点贪心当前的坐标生成结果。
- 在 `assignDepthLayers(placed)` 后新增作业顺序重排。
- `workStep` 按最终坐标的深度优先顺序重新编号：`x` 从小到大，再 `y`，再 `z`。
- `workSteps` 继续从 `PlacedBox.workStep` 派生，确保回放、分层查看和导出作业步骤读取同一套顺序。

### 5. 下一步计划

1. 运行完整单元测试，确认 `workStep` 重排没有破坏装载模式、分层统计、导出和回放。
2. 运行 `npm run lint` 与 `npm run build`，确认类型和静态检查通过。
3. 若完整测试通过，再补一次针对快照文件的人工核验：确认 `x=0,z=1800` 顶部补装箱不再晚于 `x>0` 外侧深度箱。
4. 后续如果要进一步优化实际几何排布，再单独 review 极点评分函数；本轮不把几何算法改动和作业顺序修复混在一起。

## 第二十八轮 Review 与「真实 3D 旋转」标签重构计划

本轮 review 直面一个反复回归的事实：标签旋转从第二十五轮到第二十七轮被「修」了四次又坏四次，根因不是某一处取值写错，而是整套模型选错了。用户本轮的提示一针见血——**应该按照旋转轴的角度来考虑**，而不是逐面去查角度表。

### 1. 根因：用「逐面 2D 平面角」假装 3D 旋转

问题定位：

- 现状 `labelRotationForManualFace(box, face)` 把朝向拆成 `top→yaw`、`side→pitch`、`front→0` 三个独立的 2D 平面角，箱体始终轴对齐渲染，只是交换长宽高，再给每个面贴一张 canvas 旋转过的贴图。
- 这在数学上无法表达真实旋转：绕任一轴转 90° 会同时重定向 **6 个面里的 4 个**，但现有模型每次只动「垂直于某一根轴的 2 个面」，且对 `yaw + pitch` 的复合旋转没有定义。
- 这正是二十五/二十六/二十七轮每轮都「修好又坏」的原因：在一个不完备的模型上调参，换个旋转组合就再次暴露。
- `ContainerScene` 的盒子是按交换后的尺寸建几何体，没有真正的旋转变换，所以面与贴图的对应关系本质上是猜的。

修复目标：

- 把旋转当作**真实的 3D 变换**：盒子按**原始** L/W/H 建几何体，用已有的 `orientationAxes`（signed body axes）推出旋转矩阵/四元数施加到 mesh 与 edges 上。
- 旋转后每个面只需要一张「正立」的标签贴图，物理旋转会自动把字母带到正确的面与角度——对任意 `yaw + pitch` 复合旋转都成立，by construction。
- 标签是贴在货物上的贴纸，跟随箱体一起转（用户本轮确认），方向本身即可读出朝向。

### 2. 把朝向数学收敛到单一可测模块

问题定位：

- 朝向相关计算目前散落在 `manualPlacement.ts`、`ContainerScene.tsx`、`ManualPlacement2D.tsx`、`ContainerPlan2D.tsx`，且自动装箱产出的 `PlacedBox` 只有 `orientationKey` / `labelRotationDeg`，没有 `orientationAxes`。
- `ContainerPlan2D` 的 front/side 投影一直错误地套用箱体级 `labelRotationDeg`，与手动 2D 行为不一致。

修复目标：

- 新增并单测 `src/lib/orientationTransform.ts`，作为朝向数学唯一来源：
  - `orientationAxesOf(box)`：缺省时由 `orientationKey` 推出 canonical signed axes（兼容自动装箱箱体）。
  - `baseDimensionsFromPlaced(...)`：从放置尺寸逆推原始 L/W/H，供 3D 按原始尺寸建几何体。
  - `orientationBasisVectors(axes)`：返回真实旋转的三组基向量（纯数值，不依赖 Three，便于单测）。
  - `faceLabelRotation(axes, view)`：由旋转轴几何推出 `0|90|180|270` 的面级角度，复现现有 2D 断言并对复合旋转正确。
- `ManualPlacement2D` / `ContainerPlan2D` 统一改用 `faceLabelRotation(orientationAxesOf(box), view)`，三视图共享同一规则；`data-face-label-rotation` / `data-label-rotation` 输出保持兼容。

### 本轮验收清单

1. 先写单测覆盖朝向数学：identity（LWH）、`R` 四步循环、`Shift+R` 四步循环、复合旋转、原始尺寸逆推。
2. `ContainerScene` 改为真实 3D 旋转 + 单面正立标签，移除 `labelRotationForManualFace` 依赖；确认拾取/拖拽/ghost/hover 不受影响。
3. 2D 三视图（手动 + 自动）改用 `faceLabelRotation`，保持 `data-*` 断言兼容（含 `container-calc.spec.ts:562` 的 WLH→90）。
4. `decision.md` 记录模型变更（逐面角 → 真实旋转），更新 `CHANGELOG.md` 与通知栏。
5. 本地验证：相关单元测试、`npm run lint`、`npm test`、`npm run build`、`npm run test:e2e`。
6. 范围之外：`packing.ts` 不改（自动箱体在消费端推 canonical axes）；旋转 reducer（`rotateBoxRight90/Down90` 与 `orientationAxes`）已正确，保持不动。

## 第二十七轮 Review 与物理面级标签旋转计划

本轮 review 重新定义标签旋转的物理语义：不能把一个箱体级标签角度套到六个面上。箱体绕哪个轴转，只有被该轴影响的面上的贴纸方向应该变。

### 1. 左右旋转 `R` 只改变上下两面标签方向

问题定位：

- 第二十六轮把 `labelRotationDeg` 当作整箱标签角度，导致 3D 六个面共用同一个旋转贴图。
- 物理上 `R` 是绕世界 Z 轴水平旋转：箱体顶部/底部平面上的贴纸方向会跟着转，但前后/左右竖直面贴纸不应该因为这个动作统一侧躺。
- 当前 `ContainerScene` 的 `BoxGeometry` 只挂了一个 material，无法表达不同面不同贴纸角度。

修复目标：

- `labelRotationDeg` 保留为水平上/下面的 yaw 角，用于俯视图和自动 2D 平面。
- 3D 改为按面生成材质：上/下面使用 yaw，左右面使用 pitch，前后面保持默认角度。
- 单元测试覆盖：`R` 四步循环只改变 top-face 角度，不再把 pitch 混进 top-face。

### 2. 上下旋转 `Shift+R` 只改变两侧面标签方向

问题定位：

- 第二十六轮将 `yawQuarterTurn + pitchQuarterTurn` 合并成一个 `labelRotationDeg`，所以 `Shift+R` 会错误影响俯视/上下平面标签。
- 物理上 `Shift+R` 是翻转箱体：用户关注的是两侧竖直面贴纸随翻转转 90/180/270/360，顶部/底部贴纸不应因此变化。
- 2D 前/侧视图也需要按所看的面取不同标签角度，不能继续只在 top view 旋转。

修复目标：

- 新增 `labelRotationForManualFace(...)`，按 `top/front/side` 返回面级标签角度。
- `ManualPlacement2D` 暴露 `data-face-label-rotation` 并按视图使用面级角度：俯视看 top，侧视看 side，正视看 front。
- 3D 六面材质与 2D 投影视图共享同一面级角度规则。

### 本轮验收清单

1. 先写红测覆盖物理面级旋转：`Shift+R` 不改变 top-face，`R` 不改变 front/side-face。
2. 修复 `manualPlacement`、`ManualPlacement2D`、`ContainerScene` 的面级标签角度。
3. 通知栏追加本轮变更。
4. 本地验证：相关单元测试、`npm run lint`、`npm test`、`npm run build`、`npm run test:e2e`。

## 第二十六轮 Review 与标签方向/吸附设置拆分计划

本轮 review 聚焦两个问题：标签视觉旋转仍没有表达用户所说的“A 尖头方向循环”，以及全局设置里排布设置与吸附设置混在一起，缺少总吸附开关。

### 1. 标签本体必须随 `R` / `Shift+R` 做四方向视觉循环

问题定位：

- 第二十五轮把 `labelRotationDeg` 统一置为 0，保证文字可读，但这也让非对称标签（例如 A）不会随旋转改变尖头方向。
- `orientationAxes` 已能表达空间姿态，但用户要看的不是只有轴文字，而是标签图形本身应当上、左、下、右循环。
- 3D 贴图 `makeFaceLabelTexture(...)` 和 2D `ManualPlacement2D` 都消费 `labelRotationDeg`；只要底层每次旋转更新该字段，2D/3D 可以一致显示。

修复目标：

- `R` 与 `Shift+R` 每按一次都让 `labelRotationDeg` 按 0 → 270 → 180 → 90 → 0 循环。
- 标签本体跟随旋转，朝向说明 `X:/Y:/Z:` 继续保持可读。
- 单元测试覆盖四步循环和 `toPlacedBoxes` 元数据传递；E2E 继续断言旋转后的标签方向属性。

### 2. “排布设置”和“吸附设置”必须独立，吸附要有总开关

问题定位：

- `Workbench.tsx` 目前在一个 `placement-settings-panel` 中同时放了网格/边缘/表面/Z 吸附、边缘容差、步长、悬空支撑等字段。
- 用户语义上把“排布/支撑规则”和“吸附交互规则”看成两类设置，混在一起导致入口不可预期。
- 当前只有单项吸附开关，没有一个“关闭全部吸附”的总开关；2D/3D 也需要统一尊重该开关。

修复目标：

- 工作台菜单拆成两个入口：
  - `排布设置`：部分悬空、最低支撑、提示阈值、重置。
  - `吸附设置`：总吸附开关、网格吸附、边缘吸附、上表面吸附、Z 轴吸附、步长和容差。
- `PlacementSettings` 增加 `snapEnabled`，默认 true，兼容旧 localStorage 数据。
- 总吸附关闭时，2D `applyManualPlacementSnap` 和 3D `ContainerScene` 的网格/边缘/Z/表面吸附都停止。
- E2E 验证两个面板独立、总开关关闭后 scene 暴露 `data-grid-snap="off"` / `data-edge-snap="off"`。

### 本轮验收清单

1. 单元测试覆盖标签方向循环和总吸附开关。

## 第二十五轮 Review 与旋转/标签/通知栏/画布按钮修复计划

本轮 review 聚焦四个明确问题：双轴旋转仍不符合用户意图、标签无法从当前状态识别真实旋转、站内“新特性”应改名为“通知栏”并随每次改动更新、最大化按钮仍未放到真正画布内且遮挡文字。

### 1. `R` / `Shift+R` 必须是两个方向的连续四步旋转

问题定位：

- `src/lib/manualPlacement.ts` 目前仍用 `HORIZONTAL_ROTATION_NEXT` / `DOWN_ROTATION_NEXT` 做尺寸排列跳转。
- 这只能描述 `L/W/H` 三个尺寸轴在世界 `X/Y/Z` 上的排列，无法描述同一尺寸排列下的正反朝向。
- 连续按 `R` 时，第二步和第四步都回到 `LWH` 尺寸，但它们不是同一个姿态；当前模型只靠 `yawQuarterTurn` 计数补充，标签和其他模块仍主要看 `orientationKey`，所以用户无法判断“转了一圈”是否真正回到原样。
- `Shift+R` 同样只靠二态尺寸切换，缺少完整 signed orientation 状态；后续组合旋转也会丢失顺序信息。

修复目标：

- 引入 signed orientation axes：记录 `L/W/H` 三个货物本体轴分别朝向世界 `+X/-X/+Y/-Y/+Z/-Z`。
- `orientationKey` 改为从 signed axes 派生，仅用于尺寸、碰撞和边界。
- `R` = 绕世界 Z 轴向右 90 度；连续四次必须回到完全相同的 signed axes。
- `Shift+R` = 绕世界 X 轴向下 90 度；连续四次必须回到完全相同的 signed axes。
- 单元测试覆盖四步循环、组合旋转、尺寸派生和回到原样。

### 2. 标签必须直接表达当前识别效果，不再使用 `H/I`

问题定位：

- 当前 `orientationLabel` 是 `H0 I0` / `H90 I90` 这类角度字符串。
- 用户反馈 `H/I` 本身看不出差异，也不能通过任一当前标签判断箱体究竟怎么转了。
- 3D/2D/精确面板只是显示这个字符串或 `orientationKey`，没有表达“长边/宽边/高边现在分别朝哪一个世界方向”。

修复目标：

- 废除 `H/I` 角度标签。
- 标签改为 signed axes 摘要，例如 `X:L+ Y:W- Z:H+`：从单个标签即可知道当前 X/Y/Z 三个方向分别对应货物的哪个本体轴以及正反。
- 精确面板展示更完整的三轴识别信息；2D/3D 小标签保持货物字母可读，朝向作为角标/底部标记显示。
- E2E 覆盖连续旋转后 `manual-orientation-marker` 和精确面板不再出现 `H/I` 字符串。

### 3. “新特性”改名为“通知栏”，且每次改动都必须更新

问题定位：

- `src/components/ReleaseNotesButton.tsx` 的中文入口和弹窗标题仍叫“新特性”。
- `src/data/releaseNotes.ts` 是面向用户的通知源，但第二十四轮改动没有追加通知条目。

修复目标：

- 中文入口和弹窗标题改为“通知栏”；英文改为 `Notifications`。
- 本轮在 `releaseNotes` 顶部追加 2026-05-28 通知，说明旋转、标签、通知栏命名和最大化按钮修复。
- E2E 改名为“通知栏按钮显示未读红点，点击后已读”，并断言按钮文本。

### 4. 最大化按钮必须放到真正画布内，不能遮挡工具条文字

问题定位：

- 第二十四轮把 `maximize-workspace` 放到 `visual-workspace-canvas` 右上角。
- 但手动模式下 `visual-workspace-canvas` 仍包含手动工具条、待放置池和精确面板，按钮实际覆盖的是工作区外壳，不是真正的 `ContainerScene` / `ManualPlacement2D` 画布。
- 因此按钮仍可能与手动工具条文字重叠，违背“放画布上”的要求。

修复目标：

- 删除外层 `visual-workspace-canvas` 上的最大化按钮。

  - 手动模式：`manual-view-container` 右上角。

  - 自动 2D：`ContainerPlan2D` 所在 SVG 区域右上角。
- E2E 断言按钮位于 `manual-view-container` / 自动画布区域右上，而不是外层工具条或工作区外壳。

## 第二十四轮 Review 与视图一致性修复计划

本轮 review 聚焦“所有视图行为一致”、用户提供 debug 快照中的 D 箱体拖到地面失败、最大化入口位置，以及吸附配置入口归位。

### 1. 3D 拖动 D 到地面后被误判悬空

问题定位：

- `src/components/ContainerScene.tsx` 的 plane 拖拽预览已经通过 `resolveDropTarget(...)` 计算出新的 `(x, y, z)`，当鼠标指向地面时 `z=0`。
- 但 `onPointerUp` 在 plane 模式提交时只调用 `onManualMoveRef.current?.(boxId, finalXmm, finalYmm)`，没有提交 `finalZmm`。
- `src/Workbench.tsx` 的 `manualSetBoxPosition(draft, id, x, y, z?)` 在 `z` 未传入时会保留原 `z`。
- 因此从快照中的上层 D（原 `z=600`）拖到地面时，提交结果变成“新的 X/Y + 旧的 z=600”。底下没有足够支撑后，`validateDraft(...)` 正确报出 floating，于是用户看到“操作未生效：货物处于悬空状态，底面至少需要 50% 支撑。”

修复目标：

- 3D plane 拖拽提交必须携带预览/松手时计算出的 `z`，包括从堆叠层落到地面。
- 增加回归测试，验证“原本在 `z=600` 的箱体移动到地面时，提交必须把 `z` 清为 0”，避免再次只提交 X/Y。

### 2. 2D / 3D 吸附行为不一致

问题定位：

- 3D 手动拖拽使用 `gridSnapEnabled`、`gridStepMm`、`edgeSnapEnabled`、`edgeToleranceMm`、`surfaceSnapEnabled`、`zSnapEnabled`。
- 2D 顶视图的 `ManualPlacement2D` 目前直接把 pointer 坐标传给 `onMoveBox` / `onDropFromPool`，没有应用网格吸附和边缘吸附。
- 2D 前视/侧视仍是只读投影视图，这一点符合当前设计；一致性要求应覆盖可编辑的顶视图。

修复目标：

- `ManualPlacement2D` 增加 `placementSettings` 输入，在顶视图移动和从 pool drop 时使用与 3D 相同的 X/Y 吸附顺序：先边缘吸附，再网格吸附。
- 2D 吸附只改变 `x/y`，不做表面/Z 吸附；堆叠和 Z 调整继续在 3D 或精确面板中完成。
- 增加组件测试，验证 2D 顶视图会使用 `gridStepMm` 与 `edgeToleranceMm`。

### 3. 最大化工作区入口位置不符合画布语义

问题定位：

### 4. 边缘吸附与吸附参数应放入全局设置

问题定位：

### 1. UI 风格与 archive 目标不一致

问题：

### 2. 3D 浏览缺少明确的自由视角入口

问题：

- 当前 3D canvas 支持鼠标拖拽和滚轮缩放，但界面缺少显式“自由视角”按钮。
- 用户期望有一个类似“小手”的按钮，用于进入或表明自由浏览模式。
- 当前固定视角按钮包括轴测、俯视、正视、侧视，但自由视角没有一等入口。

下一阶段要求：

- 在 3D 工具栏增加自由视角按钮，优先使用手形图标或明确的手形视觉。
- 点击自由视角后，用户可以拖拽旋转、滚轮缩放、保持当前观察目标。
- 固定视角与自由视角状态要互斥且有明确选中态。
- E2E 需要验证按钮存在、可点击、进入自由视角后 canvas 可交互且非空白。



### 3 . “装箱报告 / 货物项目 / 货柜空间”没有正常工作

问题：

- 顶部导航项目前只是静态按钮。
- “装箱报告”“货物项目”“货柜空间”没有切换视图、滚动定位、打开面板或过滤工作区的行为。
- 这会让用户误以为系统存在多模块导航，但点击没有效果。

为什么 E2E 没测出来：

- 当前 E2E 主要验证这些文字可见，没有验证点击后的状态变化。
- 测试没有定义导航项的业务含义，因此无法发现“可见但不可用”。

下一阶段要求：

- 对顶部导航做产品定义：
  - 装箱报告：切换到结果报告区域或报告页签。
  - 货物项目：聚焦或切换到货物录入清单。
  - 货柜空间：聚焦或切换到柜型参数区域。
- 如果采用 archive UI，导航可以改造成 archive 风格的主区域卡片、标签页或按钮组。
- E2E 必须点击每个导航项，并断言对应面板、焦点、选中态或 URL/状态变化。

### 5. 装载规则需要可选

问题：

- 当前“装载规则”更像说明文字，不能真正选择或影响计算。
- archive 中存在更多规则输入，例如混装排序、装载模式、承载规则、层级透明度、查看层级等。

下一阶段要求：

- 将装载规则从静态说明改为可选配置。
- 首批只实现能被当前算法支持的规则，避免 UI 承诺超过计算能力。

# 第三轮 Review 与下一阶段重构计划

日期：2026-05-20

## Review 重点

本轮 review 主要关注项目与方案保存机制、分层装载视角的物理意义修正、默认参数标准化、左侧参数栏的极限制折叠交互以及 Excel 导入的多字段弹性映射能力。下一阶段重构需要彻底解决这些深度的交互与业务逻辑一致性问题。

### 1. 关于分组与项目名称管理（右上角项目名与保存机制）

**问题**：
- 目前系统的项目或方案管理概念模糊，右上角缺少一个清晰的“项目名”作为独立实体进行生命周期管理。
- 缺少完整的新建项目、上传项目文件、保存项目文件等持久化保存机制，导致用户无法在不同设备间迁移或手动归档其装箱方案。

**下一阶段要求**：
- 在界面右上角显式提供一个**项目名称**显示与编辑区域。
- 围绕该项目名，实现以下三个核心操作按钮/功能：
  - **新建项目**：重置所有输入（柜型、规则、货物列表）并清空 3D 渲染，开启一个全新项目。
  - **保存项目**：支持将当前项目的完整配置（项目名、柜型、规则、录入货物、装载结果等）打包成一个 JSON/特定格式的本地文件，触发浏览器下载保存。
  - **上传项目**：支持用户上传之前保存的项目文件，完整还原所有参数和 3D 装箱状态。
- E2E 测试需要验证：
  - 点击“新建项目”会重置工作区。
  - 点击“保存项目”能正确触发文件下载。
  - 上传一个有效的项目文件后，参数、货物列表和 3D 图能正确加载恢复。

### 2. 本地历史记录与最近项目恢复（记住最近 5 个项目）

**问题**：
- 现有的历史记录混淆了“方案”与“项目”概念，且缺乏与服务器/本地存储的清晰结合。
- 用户需要能够直接通过最近项目列表进行一键恢复，以提高高频调整时的容错率。

**下一阶段要求**：
- 明确历史记录保存的实体就是完整的**项目**数据（包含所有参数和配置）。
- 在服务器/本地记住最近使用的 **5 个项目**（按修改/使用时间排序）。
- 在“历史方案/历史项目”入口中直观展示这 5 个最近的项目，并支持一键恢复。
- 新增第 6 个项目时，自动淘汰最早的那个项目，始终保持 5 个名额。
- E2E 验证：连续创建并保存 6 个不同的项目，确认列表中只保留最新的 5 个，并且点击其中任意一个能正确恢复所有的参数与 3D 状态。

### 3. 分层装柜作业视角重构（向内算一层，而非向下）

**问题**：
- 目前的分层查看主要是简单按 `z` 轴（高度）进行横切过滤。
- 真实的货柜装箱作业是**从内到外**（即靠集装箱里侧往外侧/门口方向）逐层堆叠和塞入的。简单的 `z` 轴过滤不符合工人实际装箱与复核的物理顺序。

**下一阶段要求**：
- 重构分层逻辑：将“层”的定义从“向下/向上（高度 z 轴）”修改为“**向内（深度 y 轴或 x 轴，取决于货柜朝向，通常以靠里为内，靠门为外）**”。
- 每一层的过滤和步进展示，应表达货物如何从货柜最里侧（底层/内侧）开始逐层堆叠，然后一步步向外塞满。
- UI 上的分层查看滑块/控件需要与此“从内向外”的真实堆叠层物理逻辑完全对齐。
- E2E 与单元测试需要验证：
  - 分层计算逻辑以货柜深度方向为基准进行分层划分。
  - 调整分层滑块时，3D 视图中货物是按“从内向外”的物理顺序被过滤或渲染出来的。

### 4. 默认货物参数标准化

**问题**：
- 现有的货物添加模板中，默认的第一个货物尺寸是 `600 * 400 * 350` mm，第二个是 `800 * 500 * 450` mm。
- 尺寸不一致且杂乱，增加了用户每次手动录入或测试时的心智负担。

**下一阶段要求**：
- 将所有新增货物的默认尺寸统一标准化为：**`400 * 500 * 600` mm**。
- 无论用户添加多少种货物，未修改前它们的初始长宽高参数必须全部为 `400 * 500 * 600` mm。
- 单元测试与 E2E 测试需要验证：点击“添加货物”按钮后，新生成的货物行中长、宽、高输入框的初始值必须为 400, 500, 600。

### 5. 左侧参数栏向左完全折叠（极限折叠与 3D 空间最大化）

**问题**：
- 在 3D 渲染和复核阶段，左侧参数栏仍然占据相当大的屏幕空间，导致 3D 视口不够开阔。
- 之前虽然提到了卡片折叠，但用户需要更彻底的“左侧栏整体向左极限制折叠”。

**下一阶段要求**：
- 支持左侧参数控制栏**整体向左完全收起/折叠**。
- 收起折叠后，左侧栏只保留极窄的一条（或仅保留三角形折叠/展开箭头图标按钮），用于点击后一键变回来。
- 左侧栏完全折叠后，右侧的 3D/2D 渲染视口和报告区域必须**自适应拉伸并占据全部/绝大部分屏幕空间**，达到沉浸式 3D 复核体验。
- E2E 验证：
  - 点击左折叠按钮后，左侧栏宽度降至极低，三角形箭头按钮仍可见且可点击。

### 5. 装箱算法不支持侧放/倾倒旋转，空间利用率严重不足

**现状**：
- 当前 `orientations()` 函数（[packing.ts](file:///home/bender/projects/cargo2/cargo_project/src/lib/packing.ts#L30-L50)）在 `canRotate = true` 时，仅生成 **2 种朝向**：原始方向和长宽互换。高度维度始终保持不变。
- 这意味着一个 `400 × 500 × 600 mm` 的箱子永远以 **600mm 作为高度**放置，从不考虑将 400mm 或 500mm 作为高度（即"侧放/倾倒"）。
- **实际影响**：
  - 默认 80 件 `400 × 500 × 600` 货物在标准 40HQ 柜（内高约 2600mm）中：
  - 当前算法：高度 600mm → 最多叠 `⌊2600/600⌋ = 4` 层，实际可能只放 3 层。
  - 允许侧放（400mm 作高度）：最多叠 `⌊2600/400⌋ = 6` 层，至少可以放 5 层（400 × 5 = 2000mm）。
  - 空间利用率差异巨大：3 层 vs 5 层意味着**上层约 40% 的有效高度被完全浪费**。
- 根本原因：`orientations()` 函数缺少对 6 种完整朝向（3 个轴 × 2 种旋转）的支持。

**下一阶段要求**：
- **修改 `orientations()` 函数**：当 `canRotate = true` 时，生成完整的 **6 种朝向**（去重后）：
  ```
  (L, W, H)  (W, L, H)    — 原始 + 水平旋转
  (L, H, W)  (H, L, W)    — 侧放（原宽度作为高度）
  (W, H, L)  (H, W, L)    — 倾倒（原长度作为高度）
  ```
- **优化放置策略**：在尝试放置每个箱子时，遍历所有有效朝向，优先选择能最大化堆叠层数或最小化浪费高度的朝向。
- **可选：新增 `canTilt` 属性**：如果业务上需要区分"水平旋转"与"侧放/倾倒"，可以增加独立的 `canTilt`（允许侧放）布尔属性。如果不需要区分，则 `canRotate = true` 同时涵盖所有 6 种朝向。
- **增加更多测试数据集**：
  - 当前 `packing.test.ts` 的测试用例偏少且场景单一，需要补充以下类型的数据集：
    - **侧放优化场景**：验证 400×500×600 箱子在 40HQ 柜中允许侧放后能叠 5+ 层。
    - **混合尺寸场景**：不同规格货物混装，验证算法能为每种货物选择最优朝向。
    - **不可旋转场景**：`canRotate = false` 的货物必须保持原始朝向，验证约束生效。
    - **边界填充场景**：验证算法在接近满载时仍能有效利用剩余空间。
    - **大量货物场景**：50-100 件货物的性能和正确性测试。
  - 单元测试验证：
  - 80 件 `400×500×600` 货物在 40HQ 柜中放置后，实际层数 ≥ 5。
  - 验证 `orientations()` 在 `canRotate = true` 时返回去重后的完整朝向列表。
  - 验证侧放后的箱子仍满足支撑关系 and 不重叠约束。

# 第七轮 Review 与下一阶段 3D 视图、标签朝向和 Excel 导入体验重构计划

日期：2026-05-22

## Review 重点

本轮 review 聚焦当前工作台最影响实际复核体验的四类问题：浏览器最大化后 3D 视图没有同步扩展，自由视角下局部箱体或面片会随视角缺失，旋转货物的标签没有跟随朝向旋转，以及 Excel 智能字段映射弹窗缺少源数据预览和单位确认能力。结合这些问题，本轮先更新 `PRD.md`，再按 PRD 对当前代码做小步、可验证的重构。

### 1. 3D 显示没有随浏览器最大化同步缩放

**现状**：
- `ContainerScene` 内部已有 `ResizeObserver`，但外层视觉工作区仍使用固定比例和最大高度约束。
- 当浏览器窗口最大化时，右侧视觉区域变宽后，3D 画布高度仍被 `aspect-[16/9]` 和 `max-h-[70vh]` 限制，无法形成大视角复核空间。

**要求**：
- 3D 工作区应优先占用可用视口高度和宽度，浏览器 resize、最大化、侧栏折叠/展开后，WebGL canvas 必须重新设置尺寸。
- 2D 视图可以继续使用稳定比例，但 3D 视图应使用更适合复核的大画布布局。
- E2E 需要验证不同 viewport 下 `canvas.width / canvas.height` 与 DOM 尺寸同步变化。

### 2. 自由视角下显示经常缺失几块

**现状**：
- 自由视角旋转时，部分箱体表面或箱体块会随视角出现缺失，用户难以判断实际装载结果。
- 当前箱体所有面使用透明材质贴图，在多层透明对象叠加时容易出现深度排序、背面剔除或透明写深度问题。

**要求**：
- 当前层和选中箱体保持不透明或近似不透明，非当前层弱化时应优先降低饱和度/提高透明度但保证形体完整。
- 材质必须关闭不合适的背面剔除，并对透明材质配置稳定的 `depthWrite` / `side` 策略。
- 自由视角拖拽、缩放后，E2E 需通过 canvas pixel 检查证明画布非空且仍有多色渲染结果。

### 3. 旋转货物的标签没有跟随旋转

**现状**：
- 货物虽然已经支持 6 种朝向，但 `PlacedBox` 没有记录原始长宽高到实际长宽高的映射。
- 2D 文本和 3D 贴图目前固定正向显示，用户无法通过标签判断箱体是水平旋转、侧放还是倾倒。

**要求**：
- `PlacedBox` 增加 `orientationKey` 和 `labelRotationDeg`，由装箱计算输出，而不是由 UI 组件临时猜测。
- 2D SVG 标签根据 `labelRotationDeg` 使用 `transform="rotate(...)"`。
- 3D 标签贴图根据朝向使用离散旋转，至少能区分原始摆放、水平旋转、侧放和倾倒。
- 单元测试必须证明旋转放置结果带有正确朝向元数据。

### 4. Excel 智能字段映射弹窗需要显示表格、数据、字段和单位

**现状**：
- 当前弹窗只展示字段下拉映射，无法看到原始表头、样例数据、单位判断和换算结果。
- 用户无法在确认导入前判断字段是否错配、厘米/毫米是否识别正确，风险较高。

**要求**：
- 弹窗扩展为导入确认工作台：左侧字段映射，右侧原始数据预览和换算预览。
- 尺寸字段必须支持 `mm`、`cm`、自动识别单位选择。
- 确认导入前显示预计导入行数、错误行数、警告行数和厘米换算行数。
- 导入解析逻辑继续放在 `src/lib/importCargo.ts` 等可测试模块中，UI 只负责展示和收集用户选择。

### 5. 代码重估与重构原则

**现状**：
- 当前代码已经具备 Vite + React + TypeScript、Express、SQLite、3D、2D、导入导出等能力，但工作台组件承担了较多状态和 UI 细节。

**要求**：
- 先更新 `PRD.md`，明确本轮新增需求和验收标准。
- 当前实现阶段优先修复最影响体验且风险较低的 3D 布局、渲染稳定性和标签朝向数据模型。
- Excel 导入弹窗重构作为下一阶段独立子任务推进，避免混入过大的 UI 改动。
- 保持简洁高效，不引入新的全局状态库、不替换 Three.js、不重写装箱算法主流程。

# 第八轮 Review 与下一阶段业务可用性、手动排布和运维加固计划

日期：2026-05-22

## Review 重点

本轮 review 聚焦从“可演示工作台”推进到“可真实使用和可运维系统”的缺口：已添加货物需要可编辑，真实业务 Excel 在指定柜型下必须 31 托全部装入，除自动排布外需要手动排布模式，顶部导航需要去掉无效入口，管理员要能查看登录审计，升级部署必须保留历史数据和用户信息，并且所有 E2E 测试必须根据需求定义，不能为了通过而修改测试。

### 1. 已添加货物需要支持编辑

**现状**：
- 当前货物列表支持新增、删除、拖拽排序，但已添加货物不能再次编辑。
- 用户录入尺寸、数量、重量、颜色或标签后，如果发现错误，只能删除并重新添加，容易丢失顺序和上下文。

**要求**：
- 在每个货物列表项的删除按钮旁增加“编辑”按钮。
- 点击后弹出该 item 的详情/编辑页，展示并允许修改完整参数：标签、名称、长宽高、重量、数量、颜色、允许旋转、允许堆叠。
- 保存后更新统一 `CargoItem[]` 数据源，并触发结果重新计算或明确标记“需要重新装箱”。
- 取消编辑不得改变原数据。
- 编辑后的变化必须同步影响 2D、3D、分层、明细、导出和历史方案。

**E2E 要求**：
- 新增货物后点击编辑，修改尺寸/数量/标签并保存，列表和明细同步更新。
- 打开编辑后取消，原数据保持不变。
- 编辑后导出结果包含修改后的参数。

### 2. 真实业务 Excel 必须在指定柜型下装入 31 托

**现状**：
- `test-data/excel/俄罗斯整托装柜尺寸.xlsx` 已作为真实业务夹具进入 E2E。
- 当前导入可识别 31 托，但在 `13400 * 2450 * 2650 mm` 自定义柜型下自动装箱只能装入 27 托。

**要求**：
- 该夹具是算法质量验收数据，必须在 E2E 中通过。
- 使用自定义货柜内尺寸 `13400 * 2450 * 2650 mm`。
- 导入后应识别 31 托，自动装箱至少装入 31 托，即全部装入。
- 不允许通过减少导入行、修改夹具、修改 E2E 期望、放宽越界/重叠/载重/支撑约束来通过。

**算法方向**：
- 重估当前极点贪心算法，解决过早推进深度或宽度导致空间碎片化的问题。
- 增加整托业务友好的铺排策略：按层、排、列组合搜索，同规格批量铺排，局部回溯或候选布局评分。
- 对真实业务夹具补单元级算法回归，E2E 只验证完整用户流程。

**E2E 要求**：
- 上传 `test-data/excel/俄罗斯整托装柜尺寸.xlsx`。
- 在映射确认后选择/设置 `13400 * 2450 * 2650 mm` 自定义柜型。
- 点击装箱后断言导入至少 31 托、已装入至少 31 托。

### 3. 支持人工手动排布模式

**现状**：
- 当前只支持自动装箱结果复核，用户不能手动拖拽、旋转或重新放置货物。
- 实际业务中，用户经常需要在自动方案基础上微调摆放或手工构造方案。

**要求**：
- 增加“自动排布 / 手动排布”模式切换。
- 手动模式保留当前自动方案作为初始草稿，或允许从空白方案开始。
- 增加待放置货物池：根据 `CargoItem[]` 和已放置数量显示剩余可放置块。
- 用户可以从货物池拖出新的块到 2D 俯视工作区。
- 用户可以拖拽已放置块、旋转块、删除块、撤销/重做。
- 首期以 2D 俯视交互为主，3D 同步展示复核结果。
- 所有手动结果必须继续通过边界、重叠、载重、支撑和堆叠校验。

**快捷键设计**：
- `R`：水平旋转选中货物。
- `Shift + R`：在允许旋转时切换侧放/倾倒候选朝向。
- 方向键：按一个网格单位移动。
- `Shift + 方向键`：按 10 个网格单位移动。
- `Delete`：删除选中块并返还待放置数量。
- `Esc`：取消拖拽或取消选中。
- `Ctrl/Cmd + Z`：撤销。
- `Ctrl/Cmd + Shift + Z`：重做。

**需要先设计清楚的问题**：
- 手动模式的数据模型如何和 `PackingResult` 共用。
- 多层手动堆叠如何选择当前层和支撑面。
- 手动方案保存到历史后如何恢复。
- 手动方案导出时如何标记为人工排布。

**E2E 要求**：
- 进入手动模式后，待放置池显示剩余数量。
- 从待放置池拖出一个货物并放入 2D 区域。
- 使用 `R` 旋转，方向键移动，`Delete` 删除。
- 尝试越界或重叠放置时出现错误提示且不能保存非法方案。

### 4. 顶部导航和主页文案精简

**现状**：
- 顶部存在 `EasyCargo`、货物项目、货柜空间、装箱报告等 Item，其中部分没有实质页面作用。
- 主页标题下方有“装箱计算、可视化复核、导入导出和本地历史方案”等说明性文案。

**要求**：

- 暂时移除“货物项目”“货柜空间”“装箱报告”等无实质页面切换作用的顶部 Item。
- 保留必要入口，如历史、用户状态、语言切换、退出登录等实际功能。
- 页面主标题保留“货柜排箱装柜工作台”。

**E2E 要求**：

### 5. 管理员需要查看最近登录历史、IP 等审计信息

**要求**：

**E2E 要求**：
- 普通用户登录一次。
- 管理员登录并打开用户管理。
- 断言该普通用户行展示最近登录时间和 IP。

### 6. 历史数据、用户信息和未来升级兼容

**现状**：
- 项目已经使用 SQLite 保存用户、自定义柜型和历史方案。
- 后续版本新增字段或表时，如果没有明确迁移机制，升级可能破坏或丢失生产数据。

**要求**：

- 禁止部署脚本默认删除数据库。
- README 或部署文档必须包含备份、恢复、迁移、回滚和健康检查说明。

**测试要求**：
- 准备旧 schema 测试数据库夹具。
- 启动服务后执行迁移。
- 验证旧用户、历史方案、自定义柜型仍然存在，新增字段可用。

### 7. 远程部署文档和部署脚本

**现状**：
- 现有 README 有部分手动部署命令，但缺少完整、可重复、可维护的部署脚本与运维说明。

**要求**：
- 增加部署脚本，例如 `scripts/deploy.mjs`。
- 部署脚本至少包含：本地构建、远程备份、文件同步、依赖安装、数据库迁移、服务重启、健康检查。

- 提供远程回滚步骤。
- 提供生产数据库备份路径、恢复命令和验证命令。
- 明确静态资源、Express 服务、SQLite 数据库的部署路径。

**E2E/运维验证要求**：
- 远程部署后运行 `PLAYWRIGHT_BASE_URL=http://<remote> npm run test:e2e`。
- 部署后健康检查 API 和首页都必须返回成功。
- 迁移和部署验证不得通过清空生产数据来获得绿色结果。

### 8. E2E 测试原则

本轮明确：E2E 测试必须根据需求定义，不得为了通过测试而修改测试。

执行原则：

- 测试夹具代表真实业务输入时，不得为迁就算法修改夹具。
- 不得把“31 托全部装入”改成“当前能装入 27 托”。

- 单元测试负责算法边界，E2E 负责用户流程和业务验收。

# 第九轮 Review 与下一阶段本地化、布局、手动排布闭环和用户管理修复计划

日期：2026-05-22

## Review 重点

第七轮和第八轮的成果上线后，业务侧在真实使用中发现了四个明确缺口：导入和装箱失败原因没有完全中文化、浏览器最大化时 3D 视图没有用满屏宽度、手动排布只完成 2D 首期、自动排布和手动排布没有形成业务闭环、管理员看不到部分新注册用户（`dengxbin`）。本轮 review 把这些缺口固化为下一阶段开发要求。

### 1. 导入和装箱失败原因仍残留英文

**现状**：
- `src/lib/importCargo.ts` 中 `warnings.push` 和 `errors.push` 直接写英文字面量，例如 `'Centimeter dimensions were converted to millimeters.'`、`'Missing or invalid length, width, or height.'`。
- `src/lib/packing.ts` 中 `markUnplaced` 的 `reason` 也是英文字面量，例如 `'No remaining loading space'`、`'Exceeds container dimensions'`、`'Exceeds maximum payload'`。
- 这些字面量经由 `Workbench` 的 import log 和 diagnostics 直接渲染，导致中文界面下出现：
  - `导入提醒 row 27: Centimeter dimensions were converted to millimeters.`
  - `29 未装入，原因：No remaining loading space.`
- `Workbench.failureReason` 已为旧 reason 做映射，但 `importCargo` 的 warning/error message 仍未本地化。

**要求**：
- 业务消息要么以 message key 的形式由 lib 输出，前端在展示时翻译；要么 lib 内部根据 locale 输出对应语言文本。
- 优先方案：把 lib 输出从「自然语言字符串」改为「带 code + params 的结构化消息」，由 Workbench 翻译成中英文。
  - `ImportCargoIssue` 增加 `code: string`、`params?: Record<string, string | number>` 字段，保留 `message` 字段做英文 fallback。
  - `UnplacedCargo.reason` 同样支持 code，或新增 `reasonCode` 字段。
- 中文文案需要覆盖以下场景：
  - 厘米换算（`row 27: 第 27 行已从厘米换算为毫米`）。
  - 缺少或非法的长宽高、重量、数量。
  - 默认数量为 1 的提醒。
  - 超出货柜尺寸、超过最大载重、没有剩余装载空间。
- 英文界面继续输出英文。

**E2E 要求**：
- 中文界面下导入夹具，验证导入日志和未装入诊断完全是中文，不出现英文 reason。
- 英文界面下同一流程验证英文文案仍可用。

### 2. 浏览器最大化时 3D 显示区只占中间一段

**现状**：
- `src/Workbench.tsx` 第 1261 行外层 `div className="mx-auto max-w-[1500px] p-5"` 把整个工作台限制在最大 1500px 宽。
- 第七轮虽然把 3D 容器改成 `h-[70vh] xl:h-[78vh]`，但宽度仍被这层 `max-w-[1500px]` 限制。
- 浏览器最大化或在 2K/4K 屏下，整页只占中间一段，左右两侧大片留白，3D 视野没有铺满。

**要求**：
- 工作台外层布局改为「左侧参数列固定/折叠 + 右侧 3D/报告区铺满剩余空间」。
- 简单做法：把 `max-w-[1500px]` 提升到 `max-w-screen-2xl` 或者直接去掉，并把布局栅格中 3D 列改为 `flex-1` 以填满剩余宽度。
- 进阶做法：基于 `100vw - paddings` 计算可用宽度，确保 3D 区域随浏览器宽度同步放大。
- 必须验证：
  - 1366 宽屏：当前布局基本不退化。
  - 1920 宽屏：3D canvas 宽度比当前明显增加。
  - 2560+ 宽屏：左右无明显留白，3D 区域填满空间。
- 2D 投影仍保留 `aspect-[16/9]` 以确保比例稳定。

**E2E 要求**：
- 用 Playwright 设置不同 `viewport`（1366 / 1920 / 2560），断言 3D canvas 宽度随 viewport 增长（允许小固定 padding）。

### 3. 手动排布 3D 视图缺失 / 与自动排布无闭环

**现状**：
- `placementMode === 'manual'` 当前只在 2D 俯视图渲染 `ManualPlacedBox[]`。
- 3D 视图 (`ContainerScene`) 始终接收的是 `result.placed`（自动结果），手动操作的箱体并不会在 3D 视图中同步显示。
- 手动模式入口与自动模式入口完全分离：进入手动模式后，自动排布的结果不会作为初始草稿；用户无法在自动结果基础上继续微调，必须从空白开始。
- 缺少业务闭环：装箱 → 微调 → 保存方案 / 导出 这条链路目前只能完整通过自动模式完成。

**要求**：
- 手动排布必须有 3D 显示能力：把 `ManualPlacedBox[]` 适配为 `PlacedBox[]`（补默认 `index`、`workStep`、`supportType`、`physicalLayer`、`supportedBy`），用同一 `ContainerScene` 渲染。
  - 越界、重叠的箱体在 3D 视图中以红色边框/材质警示。
  - 选中态、自由视角、固定视角和工具栏继续生效。
- 自动 → 手动联动：
  - 自动排布完成后，工具栏新增一个动作（例如「以自动结果继续手工微调」），把当前 `result.placed` 序列化为初始 `ManualDraft.boxes`。
  - 进入手动模式后保留所有已装箱体，用户可拖拽、旋转、删除、放回池中。
  - 池中数量按「`CargoItem.quantity` − 当前手动 draft 中该 `cargoId` 的数量」实时计算。
- 手动 → 自动联动（可选 P2）：
  - 提供「以手动草稿为起点重新自动排布」入口；当前阶段先保留按钮但暂不实现，记入 `decision.md`。
- 手动方案保存：
  - 保存历史方案时，区分 `mode: 'auto' | 'manual'`，恢复时正确进入对应模式。
  - 导出时在 shipment sheet 标记 `mode`。

**E2E 要求**：
- 自动排布完成后点击「继续手动微调」，验证手动模式 3D 视图显示了所有自动结果的箱体。
- 手动模式下旋转/移动一个箱体，验证 3D 同步、待放置池数量不变。
- 删除手动模式中的一个箱体，验证池数量 +1，3D 视图同步移除该箱体。

### 4. 管理员看不到部分新注册用户（`dengxbin`）

**现状**：
- 用户反馈 `dengxbin` 这个账号已注册，但管理员控制台中查询不到。
- 当前 `GET /api/users` 直接 `SELECT … FROM users` 没有任何 WHERE/LIMIT，前端 `UserManagement` 也没有过滤、搜索或分页，所以理论上不该漏；缺口可能在以下任一处：
  - 注册接口 `/api/auth/register` 实际未成功写入用户（异常被吞、unique 冲突未抛错给前端等）。
  - 数据库写入成功但 `disabled` 默认值或大小写敏感导致前端某种条件下漏渲染（虽然当前没看到这种条件，但需要 verify）。
  - 管理员账号本身没有刷新最新用户列表（前端没有 polling/手动刷新按钮）。
  - 远程数据库与本地数据库不同步（远程部署后没有迁移或被备份覆盖过）。

**要求**：
- 注册流程加固：

## Review 结论

### 1. 排布模式收敛为仅两种（auto / manual）

**现状**：
- `placementMode` 已是 `'auto' | 'manual'`，在工具栏中分为两个 archive-tab。
- 但用户进入手动模式后，3D 视图只能"选中"现有箱体（`ContainerScene` 的 `onSelectBox`），无法拖拽移动、无法旋转、也无法接受从待放置池中拖入。结果：手动模式在 3D 下名义存在但实际不可操作。
- 池中可拖箱目前只对 `ManualPlacement2D` 的 `<svg>` 区域生效；3D 视图没有任何 drag-drop 接收器，也没有键盘旋转/移动快捷键。

**要求**：
- 显式声明且仅保留两种排布模式：自动 (`auto`) 与手动 (`manual`)。
- 手动模式 3D 视图必须支持真正的操作：
  - 选中后可通过工具栏按钮"旋转"将选中箱体在水平面绕 Z 轴 90°（已有 `handleManualRotate` 但 3D 上未触发视觉变化的拖拽路径）。
  - 选中后可拖动其在容器底面 (XY) 内移动；落点经 `commitManualMove` 校验，发生越界/重叠时仍允许放下但标红。
  - 从待放置池 (`manual-pool-item`) 拖入 3D 区域时，箱体应基于落点 + container.length/width 反算 mm 坐标并新增到 draft。
  - 删除/撤销/重做等操作与 2D 保持一致（通过同一 `commit`/`undo`/`redo` 路径）。
- `ContainerScene` 不应被改造成完整 3D 编辑器；本轮只新增拖拽位移（XY 平面）+ pool drop 落点 + 选中状态联动，旋转复用现有工具栏按钮即可。Z 高度（堆叠）仍由后续轮次扩展。

### 2. 手动排布视角错误（2D 不会切换为前 / 侧视图）

**现状**：
- `ManualPlacement2D` 渲染的是固定俯视图：`viewBox = '0 0 length+pad*2 width+pad*2'`，使用 `container.length × container.width`。
- 上方工具栏的"俯视/正视/侧视"按钮 (`planViewMode`) 只在自动模式 `ContainerPlan2D` 起作用，手动模式下点击没有任何视觉变化——用户感觉"视角不正确"。
- 由此造成"手动模式无法看到货物在容器高度方向的堆叠情况"。

**要求**：
- `ManualPlacement2D` 必须响应外部传入的 `viewMode`（top/front/side）：
  - top：当前实现，XY 平面、纵轴向上=远端（width）。
  - front：从柜门外向内看，XZ 平面，纵轴=Z（height），横轴=X（length）。
  - side：从侧面看，YZ 平面，纵轴=Z（height），横轴=Y（width）。
- 拖拽时根据当前 viewMode 把 SVG 坐标反算成 (x, y, z) 中相应两个轴；本轮可先支持 top 视图的拖拽，front/side 至少能正确渲染只读视图（拖拽改 z 由后续轮处理）。
- 3D 视图的 viewMode 仍由 `sceneViewMode` 控制，不强制与 2D 联动。

### 3. 画布左上角的尺寸 badge 遮盖工具栏

**现状**：
- `src/Workbench.tsx:1783` 处的 badge 使用 `absolute left-5 top-5 z-10`，无论 placementMode 都覆盖在画布上。
- 自动模式下画布内部没有其他控件，badge 不影响。但手动模式的工具栏（undo/redo/旋转/删除/提示）紧贴 `manual-workspace` 顶部，会被这个尺寸 badge 完全遮挡。
- 文案是 `${selectedContainer.label}` + `${renderingContainer.length × width × height} mm`：尺寸数字本身已经跟随 `renderingContainer` 实时变动，但"标签"还是 `selectedContainer.label`（自定义柜型自定义命名场景下 OK），用户的诉求是 badge 内容随场景变动；这一点目前其实满足，但需 E2E 锁定，避免回归。

**E2E 没暴露的原因**：
- 现有 manual-mode E2E (`e2e/container-calc.spec.ts` 等) 只校验按钮存在性，没有断言按钮"可见且未被覆盖" (`toBeVisible` + bounding box 不与 badge 重叠)。
- 也没有覆盖"切换容器后 badge 内容是否同步变化"。

**要求**：
- badge 在手动模式下不得遮盖工具栏：可以把 badge 从画布绝对定位改成嵌入工具栏右侧（更稳），或者把 badge 在 `placementMode === 'manual'` 时下移到画布右上角 / 隐藏。优先采用"嵌入工具栏"方案，让 badge 在所有模式下都成为工具栏的一员，避免再次出现遮盖。
- badge 内容必须严格随 `renderingContainer`（即用户当前在表单中编辑的尺寸）变化，不再使用 `selectedContainer` 的旧标签字段以外的固定值。
- E2E 必须验证：
  - 手动模式下 undo/redo/旋转/删除四个按钮的 `boundingBox` 与 badge 不相交（或 badge 不存在于该位置）。
  - 修改自定义柜型的 length/width/height 后，badge 文本同步更新。

### 4. 装载规则默认值应改为"数量优先"

## Review 结论

### 1. 手动 3D 只能 XY 平移，缺少 Z 轴与快捷键

**现状**：
- `ContainerScene` 在 manualEditable 模式下，左键拖拽 box 时 raycast 投影到 y=0 ground plane，只能在 XY 平面平移。
- 想要把箱子抬起放到另一个箱子顶上（堆叠）无法操作；旋转、删除、微调依赖工具栏按钮。
- 手动 2D 的 front/side 视图当前是只读，不能拖动改 z。

**要求**：
- 手动模式下提供"垂直拖拽"模式：按住 Shift 拖动时沿 Z 轴升降（XY 锁定，鼠标 Y 像素映射到 z mm）。
- 快捷键支持（仅在 manualEditable + 有选中 box 时生效）：
  - R / Shift+R：水平旋转 box（绕 Z 轴 90°，方向相反）
  - Delete / Backspace：删除选中 box
  - 方向键 ←→↑↓：以 10mm 步长微调 X/Y；Shift+方向 = 100mm；Ctrl+方向 = 1mm
  - PageUp / PageDown：以 10mm 步长升降 Z；Shift+ = 100mm
  - Esc：取消选中
- 落地依赖 `manualPlacement.setBoxPosition` 接受 z 参数，保持向后兼容。
- `ContainerScene` 拖拽落点验证仍走 `validateDraft`，落点重叠/越界仍标红。

### 2. 从历史方案恢复后自动模式 3D 不刷新（admin 远程复现）

**根因**（已通过本地复现 + 调试日志定位）：
- `ContainerScene.tsx` 之前在**模块作用域**持有 `textureCache: Map` 和 `materialCache: Map`，所有 scene instance 共享。
- 当 `container` reference 变化（如 restorePlan 切换自定义柜型）时主 effect 会 cleanup 旧 scene 并 `renderer.dispose()`，释放 WebGL 资源。
- 但 module 级 cache 中的 `THREE.Texture` / `THREE.Material` 引用没被清理；下一次 mesh 创建时复用旧 material → 它们持有的 GPU side texture handle 在新 renderer context 上已失效 → 渲染时 box 表面材质丢失，整张 canvas 只剩背景 + grid + floor 颜色。
- 用户视觉表现：恢复后柜型尺寸正确、cargo 列表正确、统计数字正确、layer 数正确，**但 3D 场景里看不到任何箱体**。

**复现路径**：
1. 切换到自定义柜型并填尺寸
2. 加 cargo → 装箱 → 保存方案
3. 点"新建项目"重置工作台（触发 container 切回默认 20'）
4. 点历史方案 → 恢复（container 重新切回自定义）
5. 主 effect 重建 scene，box meshes 引用旧 cached material → GPU 资源失效 → 看不到箱体

**修复**（本轮已合入）：
- `textureCache` / `materialCache` 改为 `WeakMap<SceneState, Map>` per-scene 实例；主 effect cleanup 时 dispose 所有 cached texture/material 并清空 cache。
- 新增 regression E2E `e2e/manual-3d.spec.ts > 从历史方案恢复自定义柜型后 3D 场景重建并显示新箱体`，通过 canvas pixel sample 验证恢复后箱体颜色出现（distinct colors >= 4）。

### 3. 缺少日志和调试入口

**现状**：
- 前端目前没有用户可见的调试面板；浏览器 console 是开发者唯一渠道，普通用户没法把现场状态发回团队。

- 出现 bug 时只能要求用户描述现象，无法快速拿到关键 state。

**要求**：
- 前端：增加 `?debug=1` query 或 `Ctrl+Shift+D` 快捷键打开调试面板，展示：
  - 当前用户、locale、placementMode、workspaceView
  - selectedContainer summary（id、label、length×width×height）
  - cargoItems 数量、result.placedCount / totalCargoCount、layers 数量
  - 最近一次 history fetch 返回的计划数 / 最新 id
  - 最近 N 条前端 console.error / warn（捕获）
  - "Copy debug snapshot" 按钮把上述信息拷贝到剪贴板
- 后端：增加 `GET /api/_debug/recent-logs?limit=200`（仅 admin 鉴权）返回最近 N 行日志；面板里 admin 一键拉取。
- 日志格式：保留时间戳 + method + path + user_id（如可解析），避免敏感字段（password / token）。

# 第十二轮 Review 与下一阶段开发计划（2026-05-23）

## 背景

## Review 结论

### 1. 「继续手动微调」与「手动排布」的关系需要从产品思维重新设计

- **现状**：自动排布完成后有「继续手动微调」按钮，进入与「直接进入手动排布」相同的 manual editable 状态；两者使用同一套 ManualDraft / undo 历史，但入口、初始 boxes、用户心智模型完全不同。
- **问题**：
  - 用户搞不清「自动→手动微调」与「纯手动」是同一个模式还是两个模式；
  - 「微调」语义暗示在自动结果基础上做小调整，但当前实现允许全量删除/拖入，与「重新排布」无差别；
  - 切回自动模式时，是否保留手动调整结果、是否丢弃，缺乏明确产品规则。
- **要求**：以 PM 视角梳理两种入口的差异（目标用户、典型操作、退出条件），给出统一或区分的设计方案，并在 UI 上让用户明确知道自己处在哪个子流程。

### 2. 手动排布时 3D 自由视角（free view）不可用

- **现状**：manualEditable=true 时切换 free view 按钮，OrbitControls 仍然停留在 manual 模式（LEFT=drag，RIGHT=rotate），无法真正自由旋转；猜测是 `[manualEditable, freeView]` useEffect 的优先级判断里 manualEditable 压过 freeView。
- **要求**：手动模式下也允许临时切换 free view 作为「只看不动」的浏览态，再切回时恢复 manual 编辑。需要明确：free view 期间是否禁用拖拽？建议禁用以避免误操作，并在 UI 上提示。

### 3. Shift+Z 轴模式缺少使用说明

- **现状**：第十一轮加入 Shift 拖拽切 Z 模式与 PgUp/PgDown/方向键快捷键，但页面无任何提示，用户不知道存在；只有 decision.md 记录。
- **要求**：在手动模式工具栏 / 帮助气泡 / 空状态提示中加入快捷键说明（i18n 中英双语），并把 Shift 修饰、方向键步长（默认 10mm，Shift 100mm，Ctrl 1mm）、R 旋转、Delete 删除、Esc 取消选中都写明。

### 4. 手动排布需要遵循物理规则——禁止悬空

- **现状**：`validateDraft` 仅检查越界与碰撞，不检查支撑。用户可以把箱子拖到 z=500 悬空。
- **要求**：每个箱子要么 z=0（落地），要么底面与下方箱子顶面接触且投影有重叠（支撑判定）。
  - 拖拽时实时校验：悬空状态用 invalid 红色提示；
  - drop 时若 invalid 则回滚到上一个合法位置（与现有越界/碰撞回滚一致）；
  - 键盘移动同样限制；
  - validateDraft 增加 floating issue 类型，i18n 文案补齐。
- **边界**：堆叠角度容差、部分支撑（投影重叠 < 100%）是否允许，需要在 decision.md 记录。建议初版要求「投影重叠面积 >= 50% 底面积」，可后续放宽。

### 5. 自动排布模式下更换货柜未刷新画布

- **现状**：自动排布完成后切换 containerId（或修改 containerOverrides），ContainerScene 主 useEffect 依赖 `[container.length, container.width, container.height, doorGap, topGap, sideGap]` 会重建场景，但 boxes 数组依然来自旧的 PlacedBox 结果，与新柜尺寸不匹配——可能溢出新柜或留下空白。
- **要求**：自动模式下，container 关键尺寸或 ID 变化时清空 placedBoxes 并提示用户「已更换货柜，请重新计算」，或者自动重新触发 packing。需评估两种交互的产品效果。

### 6. PM 与用户视角的整体逻辑彻底梳理

- **现状**：经过 11 轮迭代，功能堆叠较多（auto/manual、quantity/volume/weight/input、custom container、history、validation、debug panel...），缺乏一次完整的端到端审计。
- **要求**：以「新用户首次打开」+「老用户保存方案后回访」+「admin 排查问题」三条用户故事走完整流程，记录每一步：入口是否清晰、文案是否一致、错误是否友好、状态是否正确恢复。产出一份《逻辑审计与改进列表》（建议存入 `docs/ux-audit-2026-05.md`），按优先级分批改进。

# 第十三轮 Review 与下一阶段开发计划（2026-05-23 晚）

## 背景

第十二轮上线后，手动 3D 已具备 Z 轴拖拽、键盘快捷键、物理支撑校验、自由视角只读浏览态、键盘帮助 popover、自动模式更换货柜自动清空。但用户体验仍偏「工程化」而不像「3D 建造工具」，且自由视角与拖拽完全互斥造成不便。本轮 3 条 review 直接进入实现阶段。

## Review 结论

### 1. 自由视角与拖拽的关系：用「锁定视角」按钮替代当前互斥

- **现状**：手动模式默认 LEFT=拖拽 / RIGHT=旋转 / 滚轮=缩放；点「自由视角」按钮后进入只读浏览态（manual-free），左键只能旋转、不能拖拽。两个模式互斥造成「想旋转看清角度→点自由视角→发现不能拖→再点回来→再拖」的反复切换。
- **要求**：
  - 手动模式默认即支持自由旋转视角（右键旋转、滚轮缩放、左键空白处旋转、左键箱体拖拽）。
  - 增加「锁定视角」开关：开启时所有相机控制（旋转/缩放/拖动场景）禁用，仅允许箱体拖拽，避免精细调整时手抖误旋转。
  - 移除目前「自由视角」按钮在手动模式下的语义；自由视角概念仅在自动模式下表示「解锁观察」。
  - 锁定视角按钮在手动工具栏、自动工具栏统一一致的位置 + 文案。

### 2. 3D 手动模式向 3D 建造游戏靠拢

- **现状**：拖拽时仅显示箱体本身随鼠标移动 + 实时碰撞红色，无目标位置预览、无网格吸附、无 hover 反馈、相机切换硬跳。整体像 CAD 不像 SketchUp / Minecraft。
- **要求**（本轮挑核心 3 项，其余下轮再做）：
  - **Ghost preview**：拖拽过程中显示半透明目标位置 outline（绿色合法 / 红色非法），方便用户在 release 前判断落点。
  - **Grid snap**：网格吸附开关，默认开启，步长 50mm（贴近常见集装箱单元）；关闭时回到当前自由像素拖拽。
  - **Hover highlight**：鼠标 hover 在已放置箱体上时高亮 outline + 显示 tooltip（标签/尺寸/坐标），与拖拽逻辑解耦。
- **暂不做**（记入未来 backlog）：相机缓动 lerp、撤销/重做长历史、复制粘贴/多选、剖切面视图。

### 3. 新增 PM 视角实用功能：装柜作业回放

- **现状**：自动排布生成 `workSteps`（按支撑链推算的装载顺序），目前只在「分层」「明细」表格里静态出现，工人在现场无法跟着「一步步装」。
- **要求**：
  - 新增「作业回放」面板：按 workSteps 顺序逐步在 3D 场景中显示箱体（动画 / 分步），用户可暂停、上一步、下一步、调速、跳转任意步。
  - 当前步对应箱体高亮 + 半透明预览即将放置位置；已放置箱体维持现状；未来步骤箱体隐藏。
  - 步骤列表与 3D 同步选中；点列表项跳转到对应步。
  - 回放面板可一键导出当前序列为带顺序号的作业指令 PDF / Excel（本轮先做 Excel，PDF 入下轮）。
  - 仅自动模式可用（手动模式没有 workSteps）。

# 第十五轮 Review 与下一阶段开发计划（2026-05-24）

## 背景

## Review 结论

### 1. 手动模式「贴附到表面」的拖拽（建造游戏化 v2）

- **现状**：拖动选中箱体只有两种模式 — 默认 XY 拖（z 保持原值）或 Shift+drag 切 Z（XY 锁定）。要把 A 放到 B 顶面需要先 XY 拖到 B 旁边的空位（z=0），再 Shift+drag 升上去，但这一连串步骤中途任意位置都会因为「与 B 在 z=0 重叠」而 invalid。
- **要求**：
  - 拖动一个箱体时，鼠标 raycast 不只命中地面，也命中其它已放置箱体的**顶面**。命中谁的顶面就把被拖箱体「贴」上去（z = 命中箱顶 z + 命中箱 height），ghost 同步显示。
  - 鼠标在空地（无箱命中）时维持原来的「落地或当前 z 平移」逻辑。
  - 这是建造游戏的核心交互（Minecraft 邻面 / SketchUp inferred surfaces / Lego DD「kiss」）。
  - Shift+drag 保留为「严格 Z 微调」模式（精细模式）。
  - 单元测试覆盖 `resolveDropTarget()` 工具函数：地面命中 vs 箱顶命中 vs 多个箱顶命中（取最近的）vs 命中后越界。

### 2. 明确的失败原因 + 剩余资源提示（PM 视角）

- **现状**：manualIssues 只有 boundary / overlap / floating 三种本地化文案；旋转按钮点了就转，转完才发现 invalid。用户没有看到「为什么不能」的具体数字。
- **要求**：
  - **旋转预检**：旋转前用 `manualPlacement.validateDraft({ ...candidateRotation })` 跑一遍，如果旋转后越界 / 碰撞，提示具体原因，例如：

  - **剩余资源面板**：在手动工具栏右侧 / 状态栏显示

# 第十六轮 Review 与下一阶段开发计划（2026-05-24）

## 背景

第十五轮上线了 surface-snap、旋转预检、剩余资源面板、补装建议、重心 3D。用户实测发现三个体验痛点：(a) 从左侧 pool 把新货物拖入 3D 画布时，必须松手才出现箱体——拖动过程中没有任何视觉跟随；(b) 把已放置箱体从地面往上层箱顶贴附时，拖动中 ghost 显示贴附正常，但松手后箱体又回到原位；(c) 整体手动模式仍像 CAD/调试器，远离用户对「3D 建造工具」的预期。

## Review 结论

### 1. Pool drag 进入 canvas 时立即显示 ghost

- **现状**：HTML5 native drag 从 `manual-pool` 拖到 `container-scene` 时，dragover 仅 `preventDefault()` 并设 dropEffect，没有任何 ghost 跟随。只有 `drop` 触发后才把箱体实例化进场景。
- **要求**：
  - dragstart 时把 cargo 的 length/width/height/color 信息存到 sceneState（dataTransfer 的 payload 不能在 dragover 读到）。
  - dragover 时持续 raycast (surface snap)，更新 ghost mesh 显示候选位置 + 合法/非法颜色。
  - dragleave / drop / dragend 时清理 ghost。
  - 单元测试 `resolveDropTarget` 已覆盖落点；新增 ContainerScene attribute `data-pool-ghost-active="true"` 给 E2E 断言。

### 2. Surface snap 50% 重叠检查（防止贴附后回滚）

- **现状**：`resolveDropTarget` 一旦命中其它箱顶面就贴附，但 `validateDraft` 在 commit 时要求至少 50% 底面支撑。当被拖箱比目标 surface 大时，贴附会成功显示，松手却因支撑不足回滚。用户感觉「看起来吸住了又掉下来」。
- **要求**：
  - `resolveDropTarget` 在贴附前预检：把被拖箱 XY 居中到 cursor，计算与目标 surface 顶面 XY 投影的重叠面积。若不足 `draggedArea × 50%`，再尝试「居中到 surface 中心」一次。仍不足则 **不贴附**，回退地面或被拖箱原 z（避免抖动）。
  - 补单元测试：被拖箱 1000×1000 放在 600×600 表面 → skip surface（重叠仅 36%）；被拖箱 400×400 放在 800×800 surface → 贴附（重叠 100%）。
  - 文档化：surface snap 需要至少 50% 支撑，这与 `MIN_SUPPORT_OVERLAP_RATIO` 一致。

### 3. 借鉴建造软件，对手动模式做一轮 UX 升级

- **现状**：拖动 + 键盘快捷键能用，但没有「选中后精确输入坐标」「锁定到柜面 / 柜中心」等常见辅助；用户没有反馈知道「为什么不能放」。
- **要求**（本轮做一轮可控的升级，不重写整套）：
  - **精确数值面板**：选中箱体后在工具栏右侧显示 XYZ 数值输入框，用户可输入精确 mm 值，校验通过则提交；不通过弹 rotation-notice 同款 banner。
  - **「居中」「贴墙」快捷动作**：选中后提供「居中柜底」「靠左 / 靠右 / 靠前 / 靠后」按钮，一键把箱推到对应位置（取一个 axis-aligned 角点）。
  - **当前工具栏状态显眼化**：把「已选中：标签 X、位置 (...)、尺寸 (...)」抽成 sticky 顶部 bar。
  - **拖拽过程文字提示**：ghost 旁边漂浮文字「✓ legal / ✗ overlaps box B / ✗ needs ≥50% support」(基于 computeDragInvalid 返回的具体类型)。
  - 这一阶段 **不做**：多选 / gizmo / 复制粘贴 / 二级 undo stack。等下一轮再考虑。

# 第十七轮 Review 与下一阶段开发计划（2026-05-24 续）

## 背景

第十六轮上线后用户测试发现 pool drag ghost 可见，但实际 drop 行为有两个具体 bug：
- 把货物拖到**已放置箱体上方**时，ghost 可以贴附到顶面（绿色显示正常），但松手后箱体落回地面，无法真正放到上层。
- ghost 颜色固定为绿色，不区分合法 / 不合法落点；越界、与其它箱体重叠、支撑不足时都没变红。

## Review 结论

### 1. Drop from pool 必须保留 surface snap 的 z 坐标

- **现状**：`onDrop` 仍然只 raycast `groundPlane`，丢弃了 dragover 期间 `resolveDropTarget` 得到的 z 值。`onManualDropFromPool` 的签名是 `(cargoId, x, y)`，没有 z。前端 `addBox` 默认把新箱体放在 z=0。
- **要求**：
  - `onManualDropFromPool` 扩展为 `(cargoId, x, y, z?)`，`addBox` / `manualPlacement.makeManualBox` 接受可选 z 参数（默认 0 保持向后兼容）。
  - `ContainerScene.onDrop` 用 `resolveDropTarget`，把 (x, y, z) 一起传到 Workbench。
  - 单元测试覆盖 `addBox` 带 z 的路径。

### 2. Ghost 颜色随合法性切换（pool drag + box drag 统一）

- **现状**：`dragover` 仅用 `isOutOfBounds` 判断，不检查与现有箱体的 XY 重叠、不检查支撑率（50%）。导致用户拖到「会与已有箱体冲突」的位置 ghost 仍是绿色，给出错误预期。
- **要求**：
  - 把 `computeDragInvalid` 抽成「接收 size + boxId」的通用版本（保留现有 entry-based wrapper 给 box drag 用），dragover 用通用版本判定。
  - Ghost 颜色由判定决定（绿 = 合法 / 红 = 越界 / 重叠 / 悬空）。
  - drop 时如果 ghost 是红色，拒绝放置（不调 handler）；用户得到一致反馈：ghost 红 → 松手不放 → 不会出现「我看到红的还是放下去了」。
  - E2E：构造一个拖到已有箱位置的场景，断言 ghost 期间 `data-pool-ghost-invalid="true"`。

## 下一阶段开发计划（第十七轮）

### 阶段 A：扩展 drop signature 支持 z（review #1，P0）

- `manualPlacement.makeManualBox(cargo, x, y, z?)`；`addBox(draft, box)` 保持不变（box 已含 z）。
- `Workbench.handleManualDropFromPool(cargoId, x, y, z?)` 转发 z。
- `ContainerScene` props `onManualDropFromPool?: (cargoId, x, y, z?) => void`；`onDrop` 调用 `resolveDropTarget` 并传 z。
- 单元测试：`makeManualBox` 默认 z=0 / 显式 z=500。

### 阶段 B：Ghost 红/绿 + drop 守门（review #2，P0）

- `ContainerScene` 内部把 `computeDragInvalid` 拆成 `computeInvalidByGeometry(boxId, x, y, z, l, w, h)`（无 entry），entry 版本调用 generic 版本。
- `dragover` ghost 用 generic 版本判定 invalid，positionGhost 传 invalid 颜色。
- `onDrop` 若 generic invalid → 不调 handler，clearGhost；保留 `data-pool-ghost-invalid` attribute 给 E2E。
- E2E：拖 pool 到已有箱体上 → ghost active 且 invalid=false（合法）；拖到已有箱体侧面 / 越界 → invalid=true 且 drop 被拒。

### 阶段 C：验证 + 部署

- lint/test/build/E2E + 部署 + 远程 E2E + CHANGELOG / decision.

## 执行约束

- `onManualDropFromPool` 的 z 参数必须可选，旧 callers（如果有）不破坏。
- Drop 守门是「if invalid → 不调 handler」；不要把 invalid item commit 进 draft（避免用户看到 red ghost 还放下去）。

  - **驾驶室主体**：上窄下宽的梯形外形（前部低、后部高），代表传统欧式平头卡车驾驶室。
  - **风挡**：驾驶室前面 30°-45° 倾斜的平面线框，明确指示「车头朝向」。
  - **格栅**：风挡下方、车头正前方的小竖向线段（3-4 根），仅作朝向指示。
  - **顶部导流罩（可选）**：驾驶室顶面突出 200mm 的小箱体线框，提升「卡车感」。
- 拖挂底盘：在原 deck 基础上加 **联动梁**（连接前后轴中心的纵向线段），并在 `frontAxleX` 处加 **牵引销标记**（小圆环 + 中心点）。
- 车轮：每轴左右轮维持 `TorusGeometry`，但 **每侧改为双轮组**（两个并列轮，间距 200mm），更接近实拍 HGV 后轴样貌。

**底线**：
- 不引入新依赖（不要 GLTF / OBJ / 卡车贴图）。
- 不要在 `ContainerScene.tsx` 里堆几百行 mesh 拼装逻辑——所有几何构造下沉到 `src/lib/cogVisual.ts` 的纯函数（输入 container/profile，输出几何描述对象），`ContainerScene` 只负责把描述转成 Three.js Group。

### 2. 「重心场」必须真正可见且可切换

**现状**：
- `cogVisual.ts` 没有 `buildGravityField` 函数。
- `ContainerScene` cogOverlay effect 里没有处理 `gravityFieldOn` 字段。
- `CenterOfGravityPanel` 也没有「重心场」toggle。
- 用户感受：装载重心 3D 只有一个绿/红球 + 安全范围 box，看不出「重心 + 偏离 = 风险渐变」的直观感。

**要求**：
- 在 `src/lib/cogVisual.ts` 新增 `buildGravityField(container, cog, opts?)`：
  - 返回 `Array<{ x: number; y: number; z: number; severity: number; weight: number }>`，severity ∈ [0, 1]。
  - severity 算法（最简实现）：在容器底面构造 `nx × ny` 网格点（默认 10 × 4），每个点 severity = `clamp(distanceXY(point, cog) / maxRadius, 0, 1)`；`maxRadius = sqrt((L/2)² + (W/2)²)`。
  - 边界：场点总数 ≤ 80（沿 PRD 第二十轮约束），如 container 长宽差异较大可调 nx/ny，但总数不破上限。
  - 单元测试覆盖：(a) cog 居中时所有 severity ≈ 0 在中心、≈1 在角落；(b) cog 偏向一侧时另一侧 severity 更高。
- `CogOverlay` 类型扩展 `gravityField?: GravityFieldPoint[]`；`buildCogOverlay` 接受 `opts.gravityFieldOn: boolean`，开启时填入，关闭时为 `undefined`。
- `ContainerScene` cogOverlay effect 在 `cogOverlay.gravityField` 非空时绘制场点：
  - 每个点用 **小 `SphereGeometry`**（半径 ~ container.length × 0.005，约 50-100mm）+ `MeshBasicMaterial({ color, transparent: true, opacity: 0.55 })`。
  - 颜色从 severity 0 的 `#22c55e`（绿）渐变到 severity 0.5 的 `#facc15`（黄）到 severity 1 的 `#ef4444`（红），用 HSL 插值更顺滑。
  - 全部场点放进 `state.cogGroup` 一起 dispose，关闭 toggle / 切柜型 / 解除 cogOverlay 时随 group 一起销毁。
- `CenterOfGravityPanel` 新增 toggle「重心场 / Gravity field」，并列在「在 3D 中显示」旁；状态向上 hoist 到 `Workbench`，作为 cogOverlay 的输入。
- i18n 中英文齐：`gravityFieldToggle`、`gravityFieldTooltip`。

**E2E**：
- 打开 CoG 面板 → 点「在 3D 中显示」→ 点「重心场」→ canvas 含 `data-gravity-field="on"` → 关闭 → `data-gravity-field="off"`。
- 不强求 pixel-level 测试；attribute 切换断言即可。

# 第二十二轮 Review 与下一阶段开发计划（2026-05-26）

## 背景

第二十一轮聚焦重心 3D 可视化与远程测试账号清理。本轮新的 review 转向真实装柜复核和手动排布生产力：用户需要在 3D/2D 里直接测量间距，手动排布必须支持完整六向旋转且标签能表达朝向，Excel 导入要支持用户自己的模板配置，当前“逐层添加”概念需要统一为“分层查看”，同时重心场与自动排布叠加后存在看不见或互相干扰的问题，需要重新设计显示层级。

## Review 结论

### 1. 允许放置尺规，支持测量箱体到边缘和剩余距离

**现状**：
- 当前 3D/2D 视图能显示箱体位置和容器尺寸，但用户无法直接测量某个箱体离柜壁、相邻箱或门口还有多远。
- 手动排布时虽然有 XYZ 精确输入和 hover tooltip，但它们只说明箱体自身坐标，不回答“还剩多少空间”“距离边缘多少”的现场复核问题。

**要求**：
- 增加“尺规 / 测量”工具，支持在 3D 或 2D 中选择箱体边、柜壁、相邻箱体边进行距离测量。
- 首期优先支持确定性高的测量：
  - 选中箱体后显示到左/右/前/后/顶部/门口的剩余距离。
  - 支持点击两条平行边或两个点，显示水平/垂直/空间距离。
  - 测量结果以 mm 为主，必要时补充 cm/m。
- 测量 overlay 必须是可开关的，不应永久遮挡装箱结果。
- 测量数据来自 `PlacedBox` / `ManualPlacedBox` 和 `ContainerSpec`，不能通过 DOM 像素反推。

**E2E / 测试要求**：
- 选中一个已放置箱体，打开尺规，断言出现到柜壁的距离标签。
- 拖动手动箱体后，距离数值同步变化。
- 单元测试覆盖距离计算函数，例如 `measureBoxClearance(box, container, others)`。

### 2. 手动排布支持所有方向旋转，标签必须能看出旋转

**现状**：
- 自动算法已经要求支持 6 种朝向，但手动排布仍主要是水平旋转和平面移动，立放、侧放、翻转的操作路径不完整。
- 标签旋转已有数据模型方向，但手动模式下用户无法稳定地通过标签判断当前箱体是原向、水平旋转、侧放还是倾倒。

**要求**：
- 手动排布必须支持完整 6 向旋转：
  - `LWH`、`WLH`、`LHW`、`HLW`、`WHL`、`HWL`。
  - 提供“旋转朝向”控件，优先用朝向菜单或 segmented control，而不是只靠快捷键隐藏能力。
  - `R` 保留为水平旋转；`Shift + R` 在 6 个候选朝向中循环；菜单可直接指定目标朝向。
- 旋转后必须重新校验边界、重叠、支撑、载重和不可旋转约束。
- 标签需要同步更新：
  - 2D 文本旋转角度跟随 `orientationKey` / `labelRotationDeg`。
  - 3D 标签贴图贴到当前可见主面，能区分立放、侧放和翻转。
  - hover tooltip / 明细中显示当前朝向，例如“朝向：LHW（侧放）”。

**E2E / 测试要求**：
- 手动模式选中箱体，依次切换 6 种朝向，断言尺寸投影和标签朝向变化。
- 不可旋转货物的朝向控件 disabled，快捷键不生效，并显示原因。
- 单元测试覆盖 `rotateManualBoxToOrientation` 和标签旋转映射。

### 3. Excel 支持模板导入：先配置基础模板信息，再映射字段

**现状**：
- 当前智能字段映射能处理非标准表头，但每次导入都偏一次性，用户不能把自己的业务 Excel 模板保存为可复用配置。
- 用户实际会长期使用固定的供应商模板、客户模板或内部模板，需要先配置模板基础信息，再执行字段映射。

**要求**：
- Excel 导入增加“模板导入”流程：
  - 创建/编辑模板：模板名称、适用单位（mm/cm/自动）、起始行、表头行、是否合并行、默认标签/颜色规则、数量字段规则。
  - 字段映射：将模板列映射到系统字段（标签、名称、长、宽、高、重量、数量、可旋转、可堆叠等）。
  - 预览确认：显示原始列、映射后标准字段、单位换算、错误行和警告行。
  - 保存模板：登录用户可复用；未登录场景可本地暂存。
- 模板配置必须是结构化数据，不允许用 LLM 做运行时字段判断。
- 服务器侧需要考虑用户隔离：用户 A 的模板不出现在用户 B 的模板列表中。

**E2E / 测试要求**：
- 上传一个非标准表头 Excel，创建模板并完成映射，导入成功。
- 第二次上传同模板文件时可直接选择已保存模板，字段自动映射。
- 用户隔离测试：普通用户之间模板不可见。

### 4. “逐层添加”改为“分层查看”，统一产品概念

**现状**：
- 文档和部分 UI 历史上存在“逐层添加”与“分层查看”两个概念，容易让用户误解为系统提供按层手工添加的独立流程。
- 当前真实需求是复核和表达装柜作业顺序，应统一为“分层查看”。

**要求**：
- 所有用户可见文案统一为“分层查看 / Layer view”。
- “分层查看”只表达查看、过滤、回放和复核；如果涉及手动新增箱体，应归入“手动排布”。
- 分层查看继续基于 `PackingResult.layers` / `workSteps`，不在 UI 中重新计算层。
- 如果将来要做按层手工添加，必须作为“手动排布”的子能力单独设计，不混用“分层查看”名称。

**E2E / 测试要求**：
- UI 中不再出现“逐层添加”文案。
- 分层查看 tab 可见，并能切换层级、同步 3D 可见箱体。

### 5. 重力场与自动排布显示冲突，需要重新设计

**现状**：
- 重力场 overlay 与自动排布结果同时显示时，场点、重心球、安全范围、箱体和标签会互相遮挡。
- 当自动排布箱体较多时，重力场点可能被箱体遮住，用户感觉“看不到重力场”；反过来如果重力场画在前景，又会干扰装箱复核。

**要求**：
- 重心 / 重力场需要明确显示策略，而不是简单叠加：
  - 提供显示模式：`装箱视图`、`重心视图`、`混合视图`。
  - 重心视图中箱体自动降透明或只显示轮廓，重力场和安全范围作为主视觉。
  - 装箱视图中默认关闭重力场，只保留必要的重心状态摘要。
  - 混合视图允许用户同时看箱体和重力场，但必须提供透明度滑块或强度滑块。
- 自动排布完成后进入 3D 时，默认展示装箱视图；用户打开 Balance tab 或重心开关后再进入重心视图。
- 需要保证 overlay 的 Three.js group 生命周期清晰，切换模式时不残留 mesh/material。

**E2E / 测试要求**：
- 自动排布后打开重心视图，断言 `data-cog-view-mode="cog"`，箱体透明度降低，重力场 attribute 为 on。
- 切回装箱视图，断言重力场隐藏，箱体恢复正常显示。
- 单元测试覆盖 `deriveCogViewState({ mode, gravityFieldOn, boxOpacity })`。

### 1. 手动模式拖动/放置失败时静默失败

**定位**：

- `src/components/ContainerScene.tsx:778-789`：拖拽松手后如果 `invalid`，只把 mesh 放回原位，没有通知用户失败原因。
- `src/components/ContainerScene.tsx:900-904`：从 pool drop 时如果 `finalDrop.invalid` 直接 `return`，没有触发任何 UI feedback。

**问题**：

- 用户看到的是“拖不动 / 放不上去”，系统没有说明是越界、碰撞、支撑不足、超重、不可堆叠，还是 drop 命中失败。
- 现有 `manual-issues` 只显示已存在草稿的全局问题，不能解释“这一次操作为什么没成功”。

**重构要求**：

  - 旋转失败：复用同一 notice 体系，避免只有旋转有提示。
- 反馈显示为靠近手动工作区的非阻塞 toast / status strip，不使用 `alert`，3-5 秒自动消失，并允许手动关闭。

**验收**：

- E2E 覆盖：把货物拖到柜外、拖到重叠位置、放到支撑不足的箱顶，均出现可读失败原因。
- 单元测试覆盖：从 `ValidationIssue` 到本地化提示的映射。

**重构要求**：

- 引入 `MeasurementAnnotation` 模型：
  - `id`
  - `from` / `to`：点、柜壁、箱体边、箱体角点

  - `locked`

# 第二十二轮补充 Review：旋转循环、悬空支撑、吸附配置与复核清单冲突（2026-05-27）

## 审查结论

本次补充 review 针对手动模式的新反馈做代码级定位。当前实现已经把“旋转、拖放、吸附、复核清单”做成了可用入口，但内部模型仍偏简单：旋转只保存 6 个尺寸排列，无法表达 90/180/270/360 的真实循环；支撑阈值和吸附参数全部硬编码；复核清单直接汇总合规诊断，导致两个页签职责重叠。下一步应先重构数据模型和配置层，再调 UI。

## 问题定位

### 1. `R` 旋转不是水平循环旋转，且标签朝向表达错误

**现象**：

- 用户期望 `R` 是向右 90 度循环旋转，只改变前后左右，上下不变。
- 当前操作后，类似 `G` 这种可读字母会变成侧躺状态，用户会误判箱体被侧放或倾倒。

**代码定位**：

- `src/lib/manualPlacement.ts:91-98` 的 `HORIZONTAL_ROTATION_NEXT` 只是二态互换：`LWH <-> WLH`、`LHW <-> HLW`、`WHL <-> HWL`。
- `src/lib/manualPlacement.ts:111-118` 把 `orientationKey` 直接映射到 `labelRotationDeg`，例如 `WLH = 90`。
- `src/components/ContainerScene.tsx:123-126` 在贴图内直接旋转文字，导致水平右转被表现成文字侧躺。
- `src/components/ManualPlacement2D.tsx:258-270` 同样只旋转文本，没有表达“前/后/左/右”的面朝向。

**根因**：

`orientationKey` 只描述长宽高排列，无法描述同一尺寸排列下的 180/360 朝向。例如水平旋转 180 度时尺寸仍是 `LWH`，但前后面已经交换；当前模型无法保存这个状态，只能在 `LWH` 和 `WLH` 之间跳变。

**重构要求**：

- 新增独立朝向状态，不能再只靠 `orientationKey`：
  - `yawQuarterTurn: 0 | 1 | 2 | 3` 表示水平向右循环。
  - `pitchQuarterTurn: 0 | 1 | 2 | 3` 或统一 `orientationMatrix` 表示上下循环。
  - `orientationKey` 继续作为尺寸排列的派生结果，供边界、碰撞、装箱计算使用。
- `R` 只改变水平 `yawQuarterTurn`，保持顶/底方向不变。
- 标签文字不应靠单纯旋转字形表达空间旋转。建议使用“标签文字 + F/B/L/R/U/D 或 H/I 等朝向符号 + 小型方向示意图”表达朝向；字母本体尽量保持可读。

### 2. 上下旋转也不是 90/180/270/360 循环

**现象**：

- 用户要求上下旋转同样按 90、180、270、360 循环。
- 当前 `Shift+R` 只在两个状态之间来回切换，不是完整循环。

**代码定位**：

- `src/lib/manualPlacement.ts:100-107` 的 `DOWN_ROTATION_NEXT` 也是二态互换。
- `src/lib/manualPlacement.test.ts:199-233` 当前测试反而固化了二态互换行为，因此测试没有覆盖用户真实期望。
- `src/components/ContainerScene.tsx:925-929` 和 `src/Workbench.tsx:1286-1305` 把 `Shift+R` 接到 `down`，但底层 `down` 不是四步循环。

**重构要求**：

- 用矩阵或四分之一圈状态替代 `DOWN_ROTATION_NEXT`。
- 单元测试必须覆盖：
  - 连续按 4 次 `R` 回到原朝向。
  - 连续按 4 次 `Shift+R` 回到原朝向。
  - `R` 的 4 步循环中箱体高度不变。
  - `Shift+R` 的 4 步循环中上下/前后关系按 90 度递进。

### 3. 部分悬空能力仍是硬编码，不满足可配置作业规则

**现象**：

- 用户希望上下堆叠时允许部分悬空放置。
- 当前实现看似允许部分支撑，但阈值固定为 50%，不支持按作业策略调整。

**代码定位**：

- `src/lib/manualPlacement.ts:57` 固定 `MIN_SUPPORT_OVERLAP_RATIO = 0.5`。
- `src/lib/manualPlacement.ts:275-302` 的 `findSupport` 用总重叠面积判断是否支撑。
- `src/lib/manualPlacement.ts:336-344` 低于 50% 直接报 `floating` 错误。
- `src/components/ContainerScene.tsx:593-601` 3D 拖拽预校验也硬编码同一阈值。
- `src/lib/sceneDrop.ts:64-76` 放置到上表面时，低于 50% 直接放弃吸附到上层。

**根因**：

当前系统把“结构安全硬错误”和“现场允许但需复核的悬挑”混成一个固定阈值，没有 warning/error 分层，也没有配置入口。

**重构要求**：

- 引入 `SupportPolicy`：
  - `allowPartialOverhang: boolean`
  - `minSupportRatio: number`
  - `warningSupportRatio: number`
  - `supportMode: "strict" | "balanced" | "field-review"`
- `validateDraft` 输出需要区分 `warning` 与 `error`，低于警戒线但高于最低线时允许放置并提示复核。
- 3D 预览、drop、最终校验必须使用同一个 support policy，避免预览能放、松手失败。

### 4. 吸附不可配置，且预览与最终落点存在不一致

**现象**：

- 用户要求吸附可配置，包括优化上下吸附、统一 Z 轴和左右吸附。
- 当前只有两个按钮，不能配置吸附步长、容差、Z 轴策略，也没有用户级保存。

**代码定位**：

- `src/Workbench.tsx:821-822` 只有 `gridSnap` / `edgeSnap` 两个 boolean state。
- `src/Workbench.tsx:2447-2475` UI 只有两个切换按钮，没有总体配置页或配置按钮。
- `src/lib/snap.ts:1-3` 固定 50mm 网格吸附。
- `src/lib/snapEdges.ts:3` 固定 30mm 边缘吸附容差。
- `src/components/ContainerScene.tsx:735-749` 拖拽时 XY 和 Z 分别吸附，但参数不可配。
- `src/components/ContainerScene.tsx:842-846` pool dragover 在地面 drop 时做边缘吸附；`src/components/ContainerScene.tsx:891-912` 最终 drop 重新计算后只做网格吸附，没有重复边缘吸附，可能出现预览和落点不一致。
- 当前代码里没有工作台级用户设置持久化，已有 localStorage 只用于 debug、release notes、auth token 等独立场景。

**重构要求**：

- 新增总体配置入口，建议放在工作台工具栏“设置”按钮中，而不是继续堆叠小按钮。
- 新增 `PlacementSettings`：
  - `gridSnapEnabled`
  - `gridStepMm`
  - `edgeSnapEnabled`
  - `edgeToleranceMm`
  - `zSnapEnabled`
  - `zStepMm`
  - `surfaceSnapEnabled`
  - `supportPolicy`
- 使用用户级持久化：
  - 登录用户：优先按 `currentUser.id` 保存，例如 `cargo-placement-settings:<userId>`，后续再接后端 API。
  - 未登录或 token 不可用：保存到本地默认 key，并在 UI 说明为当前浏览器设置。
- 统一 dragover、drop、键盘移动、精确面板输入和最终校验的吸附路径，不能各自算一套。

### 5. 复核清单与合规诊断职责冲突

**现象**：

- 复核清单效果不好，用户感知上和“合规与诊断”重复。

**代码定位**：

- `src/lib/reviewChecklist.ts:83-93` 直接把 `result.diagnostics` 塞进复核清单。
- `src/Workbench.tsx:2956-2965` 已有独立“合规与诊断”页签展示 diagnostics 和 unplaced。
- `src/Workbench.tsx:3052-3083` 又展示“复核清单”，其中包括 diagnostic、unplaced、manual、cog、measurement。
- `src/lib/reviewChecklist.test.ts:21-52` 测试也固化了“复核清单包含 diagnostics”的行为。

**根因**：

“合规诊断”是系统规则检查；“复核清单”应该是现场执行动作清单。当前实现把诊断结果复制到复核清单，没有重新组织成角色、动作、位置和处理建议，所以两个页签争夺同一信息。

**重构要求**：

- 保留“合规与诊断”作为规则检查页。
- 将“复核清单”重命名或重构为“现场复核 / 作业复核”：
  - 只展示需要现场处理的动作项。
  - 每条必须包含对象、位置、原因、建议动作、来源链接。
  - diagnostics 只作为来源引用，不再原样复制。
- 如果短期无法重构，建议先隐藏复核清单页签，避免制造重复信息。
