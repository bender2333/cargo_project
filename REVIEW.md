# Review 与下一阶段开发计划

日期：2026-05-20

## 背景

本次 review 针对当前重构版本与旧版参考页面的差异、3D 操作可用性、导航和表单功能完整性、装载规则可配置性，以及 E2E 测试为什么没有暴露这些问题。

参考页面：

- `archive/active-8080/index.html`

当前实现：

- `src/Workbench.tsx`
- `src/components/ContainerScene.tsx`
- `src/components/ContainerPlan2D.tsx`
- `src/lib/*`
- `e2e/container-calc.spec.ts`

## Review 结论

### 1. UI 风格与 archive 目标不一致

问题：

- 当前版本继续沿用了灰色 EasyCargo 风格工作台。
- 用户确认 `archive/active-8080/index.html` 的界面、UI 布局和按钮风格才是正确方向。
- archive 中的 3D、显示逻辑和后端脚本不能直接作为新架构依赖；但它的界面组织、按钮、卡片、标签页、工具栏和视觉风格需要迁移到当前 React + TypeScript 项目。

下一阶段要求：

- 使用 archive 的 UI 作为产品外观基准。
- 保留当前项目的装箱算法、`PackingResult`、3D/2D 数据结构、导入导出、历史方案和测试体系。
- 不把 archive 的旧版静态脚本作为运行时依赖。

### 2. 3D 浏览缺少明确的自由视角入口

问题：

- 当前 3D canvas 支持鼠标拖拽和滚轮缩放，但界面缺少显式“自由视角”按钮。
- 用户期望有一个类似“小手”的按钮，用于进入或表明自由浏览模式。
- 当前固定视角按钮包括轴测、俯视、正视、侧视，但自由视角没有一等入口。

下一阶段要求：

- 在 3D 工具栏增加自由视角按钮，优先使用手形图标或明确的手形视觉。
- 点击自由视角后，用户可以拖拽旋转、滚轮缩放、保持当前观察目标。
- 固定视角与自由视角状态要互斥且有明确选中态。
- E2E 需要验证按钮存在、可点击、进入自由视角后 canvas 可交互且非空白。



### 3 . “装箱报告 / 货物项目 / 货柜空间”没有正常工作

问题：

- 顶部导航项目前只是静态按钮。
- “装箱报告”“货物项目”“货柜空间”没有切换视图、滚动定位、打开面板或过滤工作区的行为。
- 这会让用户误以为系统存在多模块导航，但点击没有效果。

为什么 E2E 没测出来：

- 当前 E2E 主要验证这些文字可见，没有验证点击后的状态变化。
- 测试没有定义导航项的业务含义，因此无法发现“可见但不可用”。

下一阶段要求：

- 对顶部导航做产品定义：
  - 装箱报告：切换到结果报告区域或报告页签。
  - 货物项目：聚焦或切换到货物录入清单。
  - 货柜空间：聚焦或切换到柜型参数区域。
- 如果采用 archive UI，导航可以改造成 archive 风格的主区域卡片、标签页或按钮组。
- E2E 必须点击每个导航项，并断言对应面板、焦点、选中态或 URL/状态变化。

### 5. 装载规则需要可选

问题：

- 当前“装载规则”更像说明文字，不能真正选择或影响计算。
- archive 中存在更多规则输入，例如混装排序、装载模式、承载规则、层级透明度、查看层级等。
- 当前实现已有 `loadingMode`，但规则配置仍不足，且“规则说明/装载规则”没有形成可验证的业务输入。

下一阶段要求：

- 将装载规则从静态说明改为可选配置。
- 首批只实现能被当前算法支持的规则，避免 UI 承诺超过计算能力。
- 每个可选规则都必须进入 `calculatePacking` 或结果展示逻辑，并有单元测试或 E2E 证明它会改变结果或视图。
- 暂不支持的 archive 规则需要在 `decision.md` 记录暂缓原因，而不是做成无效控件。

### 阶段 4：实现可选装载规则

范围：

- 梳理 archive 中规则项与当前算法能力的对应关系。
- 优先实现当前可支撑的规则，例如装载排序、可旋转、可堆叠、有效边界、层级查看和透明度。
- 需要新增算法能力的规则先写入 `decision.md`，分阶段实现。

验收：

- 每个规则都有单元测试或 E2E 覆盖。
- 不出现“UI 可选但计算不使用”的假功能。

### 阶段 5：E2E 测试补强与防回归

目标：

- 让浏览器测试覆盖真实用户行为，而不是只覆盖元素存在。

新增测试原则：

- 不允许通过修改测试降低断言来获得通过。
- 不允许删除失败测试来掩盖功能缺陷。
- 修改 E2E 的唯一合理原因是产品行为被明确重新定义，并且同步更新 review/decision 文档。
- 每个新增控件必须至少有一个测试证明它能改变状态、结果、视图、历史或导出。
- 对“看起来像按钮/输入框/导航”的元素，必须测试点击或输入后的结果。

建议新增 E2E 覆盖：

- archive 风格关键布局存在：header、基础参数卡片、货物清单卡片、统计卡片、结果标签页、工具栏。
- 顶部导航点击后切换或聚焦对应区域。
- 装运名称保存到历史并可恢复。
- `≡` 点击后出现明确菜单或行为。
- 3D 自由视角按钮存在、选中态变化、拖拽和缩放后 canvas 非空白。
- 装载规则选择会影响计算输入或展示状态。

# 第二轮 Review 与下一阶段重构计划

日期：2026-05-20

## Review 重点

本次 review 的核心不是算法回退，而是继续把工作台从“功能可用”推进到“布局、信息层级和操作流符合装箱工具使用方式”。当前主要问题集中在左侧输入区过长、导入导出消息占用主页面、报告面板挤占 3D 空间、货物列表缺少管理能力，以及历史方案仍不像独立页面。
### 0. 默认语言修改为中文。

### 1. 左侧货柜参数需要可折叠

问题：

- 货柜参数目前始终展开，左侧信息密度过高。
- 用户常用流程里，柜型参数设置完成后主要关注货物录入和 3D 结果。

下一阶段要求：

- 货柜参数卡片支持展开/折叠。
- 折叠态保留标题、当前柜型和有效尺寸摘要。
- 展开/折叠状态必须有明确按钮、`aria-expanded` 和可见状态变化。
- E2E 需要验证折叠后参数输入不可见、摘要仍可见，再次展开后输入恢复。

### 2. 装载规则需要可折叠

问题：

- 装载规则目前仍占用左侧空间。
- 规则是重要输入，但不是每次操作都需要展开查看。

下一阶段要求：

- 装载规则卡片支持展开/折叠。
- 折叠态保留当前规则摘要，例如“体积优先 / 重量优先 / 数量优先 / 录入顺序”。
- 规则选择仍必须保持真实计算输入，不能因为折叠变成显示状态。
- E2E 需要验证折叠后仍能看到当前规则摘要，展开后切换规则并影响装柜步骤。

### 3. Excel 导入导出需要移到装箱报告附近，导入导出警告进入单独页签

问题：

- 导入 Excel 和导出 Excel 当前放在左侧货物列表附近，会和货物录入主流程混在一起。
- 导入警告、字段识别、厘米换算、错误行等消息直接挤在主页面，影响工作区清晰度。

下一阶段要求：

- 将 Excel 导入、导出放到装箱报告/结果工具栏附近，和“导出视图”同属结果操作。
- 导入状态、导入警告、导入错误、字段映射摘要放入独立页签，例如“导入日志”。
- 主页面只保留必要的简短状态提示，不能长消息堆叠。
- 导入后仍进入当前货物数据集，并能触发装箱、明细、导出流程。
- E2E 需要验证：
  - 导入控件位于报告区域。
  - 导入后主页面不出现长警告列表。
  - “导入日志”页签展示导入成功、字段识别、换算行数和警告。
  - 导出 Excel 仍包含当前装箱结果和装运信息。

### 4. 整体布局改为两列，3D 显示占当前页面约 3/4

问题：

- 最右侧的“逐层 / 明细表 / 合规与诊断 / 历史方案”面板挤占横向空间。
- 3D 显示区域没有充分占据主视野，不符合装箱复核的核心工作流。

下一阶段要求：

- 主体布局改为两列：
  - 左列：参数、规则、货物录入和货物列表。
  - 右列：顶部为 3D/2D 显示和统计，约占主内容宽度的 3/4。
- “逐层 / 明细表 / 合规与诊断”作为 3D 显示器下方的报告 panel，不再作为最右侧侧栏。
- 历史方案从该 panel 中移出，作为独立页面或独立主视图。
- E2E 需要验证不存在右侧独立报告侧栏，报告 panel 位于 3D 显示下方，3D canvas 宽度显著大于左侧栏。

### 5. 货物添加后支持删除

问题：

- 当前货物列表只支持选择，缺少删除能力。
- 用户录入错误时只能通过刷新或导入覆盖绕过。

下一阶段要求：

- 每个货物项提供删除按钮。
- 删除后 `cargoItems`、`PackingResult`、2D、3D、明细、导出和历史保存都使用删除后的数据。
- 如果删除的是当前选中箱体，需要清空或更新选中态。
- E2E 需要验证删除后货物项消失、装箱数量和货物类型数更新、导出不包含已删除货物。

### 6. 左侧货物列表支持拖拽修改顺序

问题：

- 当前“录入顺序”规则依赖货物数组顺序，但用户无法在 UI 中调整顺序。
- 这导致“录入顺序”规则实际只能按添加顺序工作，不利于装柜作业规划。

下一阶段要求：

- 货物列表支持拖拽排序，优先使用浏览器原生 drag/drop 或轻量实现，避免引入不必要依赖。
- 拖拽后更新 `cargoItems` 顺序。
- 在“录入顺序”装载规则下，排序变化必须影响 `calculatePacking` 的作业步骤。
- E2E 需要验证拖拽改变列表顺序，并在切换到录入顺序后改变第一步装载标签。

### 7. 历史方案作为独立页展示，本地保留 5 个历史项目

问题：

- 历史方案目前在结果页签中，和逐层、明细、诊断混在一起。
- review 要求历史方案成为独立页面/主视图。
- 当前存储限制已有 5 条，但需要在新页面里明确验证和展示。

下一阶段要求：

- 顶部导航或菜单提供“历史方案”独立入口。
- 历史方案页面展示本地最近 5 个历史项目。
- 保存第 6 个方案时，只保留最近 5 个，最旧项被移除。
- 恢复历史方案后返回工作台，并恢复装运名称、柜型、货物、标签、结果摘要。
- E2E 需要验证历史页独立显示、最多 5 项、恢复后工作台状态正确。

### 1. 关于分组与项目名称管理（右上角项目名与保存机制）

- 目前系统的项目或方案管理概念模糊，右上角缺少一个清晰的“项目名”作为独立实体进行生命周期管理。

**下一阶段要求**：

### 2. 本地历史记录与最近项目恢复（记住最近 5 个项目）

**问题**：

**下一阶段要求**：

### 3. 分层装柜作业视角重构（向内算一层，而非向下）

**问题**：
- 目前的分层查看主要是简单按 `z` 轴（高度）进行横切过滤。
- 真实的货柜装箱作业是**从内到外**（即靠集装箱里侧往外侧/门口方向）逐层堆叠和塞入的。简单的 `z` 轴过滤不符合工人实际装箱与复核的物理顺序。

**下一阶段要求**：
- 重构分层逻辑：将“层”的定义从“向下/向上（高度 z 轴）”修改为“**向内（深度 y 轴或 x 轴，取决于货柜朝向，通常以靠里为内，靠门为外）**”。
- 每一层的过滤和步进展示，应表达货物如何从货柜最里侧（底层/内侧）开始逐层堆叠，然后一步步向外塞满。
- UI 上的分层查看滑块/控件需要与此“从内向外”的真实堆叠层物理逻辑完全对齐。
- E2E 与单元测试需要验证：
  - 分层计算逻辑以货柜深度方向为基准进行分层划分。
  - 调整分层滑块时，3D 视图中货物是按“从内向外”的物理顺序被过滤或渲染出来的。

### 4. 默认货物参数标准化

**问题**：
- 现有的货物添加模板中，默认的第一个货物尺寸是 `600 * 400 * 350` mm，第二个是 `800 * 500 * 450` mm。
- 尺寸不一致且杂乱，增加了用户每次手动录入或测试时的心智负担。

**下一阶段要求**：
- 将所有新增货物的默认尺寸统一标准化为：**`400 * 500 * 600` mm**。
- 无论用户添加多少种货物，未修改前它们的初始长宽高参数必须全部为 `400 * 500 * 600` mm。
- 单元测试与 E2E 测试需要验证：点击“添加货物”按钮后，新生成的货物行中长、宽、高输入框的初始值必须为 400, 500, 600。

### 5. 左侧参数栏向左完全折叠（极限折叠与 3D 空间最大化）

**问题**：
- 在 3D 渲染和复核阶段，左侧参数栏仍然占据相当大的屏幕空间，导致 3D 视口不够开阔。
- 之前虽然提到了卡片折叠，但用户需要更彻底的“左侧栏整体向左极限制折叠”。

**下一阶段要求**：

- 收起折叠后，左侧栏只保留极窄的一条（或仅保留三角形折叠/展开箭头图标按钮），用于点击后一键变回来。
- 左侧栏完全折叠后，右侧的 3D/2D 渲染视口和报告区域必须**自适应拉伸并占据全部/绝大部分屏幕空间**，达到沉浸式 3D 复核体验。
- E2E 验证：
  - 点击左折叠按钮后，左侧栏宽度降至极低，三角形箭头按钮仍可见且可点击。
  - 折叠后 3D canvas 的宽度应显著增大，接近屏幕全宽。
  - 点击三角形箭头后，左侧栏能够恢复原状， 3D canvas 宽度回弹。

### 6. Excel 导入支持动态字段重新配置（智能字段映射）

**问题**：

# 第四轮 Review 与下一阶段重构计划

日期：2026-05-21

## Review 重点

本轮 review 提出了四个方向的新需求：自定义箱型的完整管理能力、用户账号系统与数据隔离、3D 视图折叠后的宽高比保持、以及后端服务的远程部署与端到端验证。其中账号系统是架构层面的重大升级，需要引入后端服务、数据库和认证机制。

### 1. 自定义箱型缺少完整的管理能力

**现状**：
- 当前系统内置了若干标准柜型（20GP、40GP、40HQ 等），并提供了一个"自定义柜型"选项。
- 选择自定义柜型后，用户可以在左侧参数区直接编辑长、宽、高、最大载重、门口间隙、顶部间隙和侧向间隙。

  - 只有一个固定的"Custom container"占位项，无法同时保留多个自定义箱型。

  - 项目保存/恢复时虽然会携带 `customContainer` 字段，但这只是单个快照，不是用户可管理的自定义箱型库。

**下一阶段要求**：

### 2. 用户账号系统与数据隔离

**现状**：

**下一阶段要求**：

    - 系统启动时自动创建默认管理员账号（如 `admin` / `admin123`，首次登录强制修改密码）。
    - 管理员可查看所有用户列表、禁用/删除用户账号。
  - **数据隔离**：
    - 每个用户独立存储：历史方案列表、自定义箱型库、项目配置。
    - 用户 A 的数据对用户 B 不可见。

  - 登录后顶部显示当前用户名和退出按钮。
  - 管理员登录后在导航中显示"用户管理"入口。

- 数据库选型建议：

- E2E 验证：

  - 不同用户登录后看不到对方的历史方案和自定义箱型。
  - 管理员登录后可见用户管理页面，可禁用/删除普通用户。
  - 未登录时访问工作台自动跳转到登录页。

### 3. 左侧折叠后 3D 视图应保持宽高比放大

**现状**：
- 左侧参数栏折叠后，右侧 3D Canvas 的 `ResizeObserver` 会检测到容器尺寸变化，并通过 `camera.aspect = mount.clientWidth / mount.clientHeight` 更新相机宽高比。
- 这导致 3D 视图**只在水平方向拉伸**，而高度保持不变，视觉上比例失调、画面变形。
- 用户期望的是：折叠后 3D 渲染区域在**保持原有宽高比的前提下整体等比放大**，视觉效果类似于"画面等比缩放"，而不是单纯的横向拉伸。

**下一阶段要求**：
- 折叠左侧栏后，3D Canvas 容器应在保持固定宽高比（例如 16:9 或当前容器比例）的前提下，在可用空间内最大化显示。
- 实现方式建议：给 Canvas 容器设置 `aspect-ratio` CSS 约束（或通过 JS 计算高度 = 宽度 / 比例），使宽高同步放大。
- 展开左侧栏时，3D 区域恢复原始尺寸。
- E2E 验证：
  - 折叠前记录 Canvas 的 width 和 height。
  - 折叠后验证 Canvas 的 width 和 height 都增大，且宽高比与折叠前基本一致（允许 5% 误差）。

### 5. 装箱算法不支持侧放/倾倒旋转，空间利用率严重不足

**现状**：
- 当前 `orientations()` 函数（[packing.ts](file:///home/bender/projects/cargo2/cargo_project/src/lib/packing.ts#L30-L50)）在 `canRotate = true` 时，仅生成 **2 种朝向**：原始方向和长宽互换。高度维度始终保持不变。
- 这意味着一个 `400 × 500 × 600 mm` 的箱子永远以 **600mm 作为高度**放置，从不考虑将 400mm 或 500mm 作为高度（即"侧放/倾倒"）。
- **实际影响**：
  - 默认 80 件 `400 × 500 × 600` 货物在标准 40HQ 柜（内高约 2600mm）中：
  - 当前算法：高度 600mm → 最多叠 `⌊2600/600⌋ = 4` 层，实际可能只放 3 层。
  - 允许侧放（400mm 作高度）：最多叠 `⌊2600/400⌋ = 6` 层，至少可以放 5 层（400 × 5 = 2000mm）。
  - 空间利用率差异巨大：3 层 vs 5 层意味着**上层约 40% 的有效高度被完全浪费**。
- 根本原因：`orientations()` 函数缺少对 6 种完整朝向（3 个轴 × 2 种旋转）的支持。

**下一阶段要求**：
- **修改 `orientations()` 函数**：当 `canRotate = true` 时，生成完整的 **6 种朝向**（去重后）：
  ```
  (L, W, H)  (W, L, H)    — 原始 + 水平旋转
  (L, H, W)  (H, L, W)    — 侧放（原宽度作为高度）
  (W, H, L)  (H, W, L)    — 倾倒（原长度作为高度）
  ```
- **优化放置策略**：在尝试放置每个箱子时，遍历所有有效朝向，优先选择能最大化堆叠层数或最小化浪费高度的朝向。
- **可选：新增 `canTilt` 属性**：如果业务上需要区分"水平旋转"与"侧放/倾倒"，可以增加独立的 `canTilt`（允许侧放）布尔属性。如果不需要区分，则 `canRotate = true` 同时涵盖所有 6 种朝向。
- **增加更多测试数据集**：
  - 当前 `packing.test.ts` 的测试用例偏少且场景单一，需要补充以下类型的数据集：
    - **侧放优化场景**：验证 400×500×600 箱子在 40HQ 柜中允许侧放后能叠 5+ 层。
    - **混合尺寸场景**：不同规格货物混装，验证算法能为每种货物选择最优朝向。
    - **不可旋转场景**：`canRotate = false` 的货物必须保持原始朝向，验证约束生效。
    - **边界填充场景**：验证算法在接近满载时仍能有效利用剩余空间。
    - **大量货物场景**：50-100 件货物的性能和正确性测试。
  - 单元测试验证：
  - 80 件 `400×500×600` 货物在 40HQ 柜中放置后，实际层数 ≥ 5。
  - 验证 `orientations()` 在 `canRotate = true` 时返回去重后的完整朝向列表。
  - 验证侧放后的箱子仍满足支撑关系 and 不重叠约束。

# 第七轮 Review 与下一阶段 3D 视图、标签朝向和 Excel 导入体验重构计划

日期：2026-05-22

## Review 重点

本轮 review 聚焦当前工作台最影响实际复核体验的四类问题：浏览器最大化后 3D 视图没有同步扩展，自由视角下局部箱体或面片会随视角缺失，旋转货物的标签没有跟随朝向旋转，以及 Excel 智能字段映射弹窗缺少源数据预览和单位确认能力。结合这些问题，本轮先更新 `PRD.md`，再按 PRD 对当前代码做小步、可验证的重构。

### 1. 3D 显示没有随浏览器最大化同步缩放

**现状**：
- `ContainerScene` 内部已有 `ResizeObserver`，但外层视觉工作区仍使用固定比例和最大高度约束。
- 当浏览器窗口最大化时，右侧视觉区域变宽后，3D 画布高度仍被 `aspect-[16/9]` 和 `max-h-[70vh]` 限制，无法形成大视角复核空间。

**要求**：
- 3D 工作区应优先占用可用视口高度和宽度，浏览器 resize、最大化、侧栏折叠/展开后，WebGL canvas 必须重新设置尺寸。
- 2D 视图可以继续使用稳定比例，但 3D 视图应使用更适合复核的大画布布局。
- E2E 需要验证不同 viewport 下 `canvas.width / canvas.height` 与 DOM 尺寸同步变化。

### 2. 自由视角下显示经常缺失几块

**现状**：
- 自由视角旋转时，部分箱体表面或箱体块会随视角出现缺失，用户难以判断实际装载结果。
- 当前箱体所有面使用透明材质贴图，在多层透明对象叠加时容易出现深度排序、背面剔除或透明写深度问题。

**要求**：
- 当前层和选中箱体保持不透明或近似不透明，非当前层弱化时应优先降低饱和度/提高透明度但保证形体完整。
- 材质必须关闭不合适的背面剔除，并对透明材质配置稳定的 `depthWrite` / `side` 策略。
- 自由视角拖拽、缩放后，E2E 需通过 canvas pixel 检查证明画布非空且仍有多色渲染结果。

### 3. 旋转货物的标签没有跟随旋转

**现状**：
- 货物虽然已经支持 6 种朝向，但 `PlacedBox` 没有记录原始长宽高到实际长宽高的映射。
- 2D 文本和 3D 贴图目前固定正向显示，用户无法通过标签判断箱体是水平旋转、侧放还是倾倒。

**要求**：
- `PlacedBox` 增加 `orientationKey` 和 `labelRotationDeg`，由装箱计算输出，而不是由 UI 组件临时猜测。
- 2D SVG 标签根据 `labelRotationDeg` 使用 `transform="rotate(...)"`。
- 3D 标签贴图根据朝向使用离散旋转，至少能区分原始摆放、水平旋转、侧放和倾倒。
- 单元测试必须证明旋转放置结果带有正确朝向元数据。

### 4. Excel 智能字段映射弹窗需要显示表格、数据、字段和单位

**现状**：
- 当前弹窗只展示字段下拉映射，无法看到原始表头、样例数据、单位判断和换算结果。
- 用户无法在确认导入前判断字段是否错配、厘米/毫米是否识别正确，风险较高。

**要求**：
- 弹窗扩展为导入确认工作台：左侧字段映射，右侧原始数据预览和换算预览。
- 尺寸字段必须支持 `mm`、`cm`、自动识别单位选择。
- 确认导入前显示预计导入行数、错误行数、警告行数和厘米换算行数。
- 导入解析逻辑继续放在 `src/lib/importCargo.ts` 等可测试模块中，UI 只负责展示和收集用户选择。

### 5. 代码重估与重构原则

**现状**：
- 当前代码已经具备 Vite + React + TypeScript、Express、SQLite、3D、2D、导入导出等能力，但工作台组件承担了较多状态和 UI 细节。

**要求**：

## 下一阶段重构执行计划（第七轮）

### 阶段 1：PRD 与 Review 同步
- **目标**：把本轮 3D、标签朝向、Excel 映射体验和代码重估要求写入产品要求。
- **范围**：
  - 更新 `PRD.md` 的标签展示、3D、Excel 导入映射、非功能需求、验收标准和测试要求。
  - 在 `REVIEW.md` 增加第七轮 review 和可执行计划。
- **验收**：文档明确新增需求、验收标准和不做范围。

### 阶段 2：3D 自适应和自由视角渲染稳定性
- **目标**：让 3D 画布在大屏/最大化下真正扩展，并修复自由视角下箱体缺失。
- **范围**：
  - 调整视觉工作区高度约束，3D 使用 viewport-aware 高度，2D 保持稳定可读比例。
  - 强化 `ContainerScene` 的 resize 逻辑，确保 `renderer.setSize` 与 DOM 尺寸同步。
  - 调整 Three.js 材质透明策略，避免局部面片因视角缺失。
- **验收**：
  - 最大化或 resize 后 3D canvas 尺寸跟随容器变化。
  - 自由视角拖拽/缩放后 canvas 仍有稳定多色渲染。

### 阶段 3：标签旋转朝向数据模型
- **目标**：让标签旋转成为计算结果的一部分，而不是渲染层猜测。
- **范围**：
  - 扩展 `PlacedBox`：增加 `orientationKey` 与 `labelRotationDeg`。
  - `orientations()` 输出朝向元数据，`calculatePacking()` 将其写入每个已装箱体。
  - 2D 文本使用 `labelRotationDeg` 做 SVG transform。
  - 3D 贴图按 `labelRotationDeg` 绘制旋转标签。
- **验收**：
  - 单元测试覆盖水平旋转、侧放/倾倒的朝向元数据。
  - 2D/3D 可通过标签方向看出旋转差异。

### 阶段 4：Excel 映射确认工作台
- **目标**：把智能字段映射从简单下拉弹窗升级为可确认、可调整单位的导入工作台。
- **范围**：
  - 抽离导入映射配置类型和单位配置。
  - 显示原始表头、前 5-10 行数据和转换后的标准字段预览。
  - 支持修改长宽高单位判断，并在预览中即时反映毫米换算。
  - 增加行级错误/警告摘要。
- **验收**：
  - 用户在确认导入前能看到字段映射、原始值、单位选择和转换后值。
  - 解析逻辑有单元测试，弹窗有 E2E 覆盖。

## 执行约束

- `archive/` 继续只作为产品参考，不能把旧版 bundle 接入新架构。
- 不在本轮引入新的 3D 引擎、状态管理库或复杂抽象。
