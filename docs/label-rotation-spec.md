# 标签旋转修复规格（交付 codex 执行）

> 目的：彻底修正手动模式 3D / 2D 中"货物旋转后标签 A 朝向不正确"的问题。
> 本文件是**需求 + 严格测试契约**。codex 可自由选择实现方式，但**必须让本文所有测试通过，且不得放松/删改本文要求的断言**。

---

## 0. 名词与坐标约定

内部单位统一 mm。数据坐标系（与 `PlacedBox.x/y/z` 一致）：

- `x` 轴：柜**长** length 方向（朝柜门外为 +）
- `y` 轴：柜**宽** width 方向
- `z` 轴：柜**高** height 方向，`z+` 朝上

箱子**本体轴 body axes**：`L` `W` `H` = 货物原始的长、宽、高三条棱（旋转不改变它们的归属，只改变它们指向哪个世界方向）。

`orientationAxes = { x, y, z }`：每个**世界轴**当前对应哪条**本体轴**（带正负号）。
基准姿态（未旋转）：`{ x: 'L+', y: 'W+', z: 'H+' }`。

`A` 标签：贴在箱子上的**物理贴纸**。箱子是刚体，旋转时贴纸随箱子一起转——这是整套需求的唯一第一性原理。

## 1. 旋转语义（已锁定，不得更改）

下列序列已由 `manualPlacement.ts` 的 reducer 与现有单测确立，**视为正确，不准动**。本次只修标签朝向，不改旋转本身。

### 1.1 R = 左右旋转（`rotateBoxRight90`），绕世界 `z` 轴（竖直）

| 步骤 | orientationAxes | orientationKey |
|---|---|---|
| 基准 | `L+ W+ H+` | LWH |
| R ×1 | `W+ L- H+` | WLH |
| R ×2 | `L- W- H+` | LWH |
| R ×3 | `W- L+ H+` | WLH |
| R ×4 | `L+ W+ H+` | LWH |

注意：`orientationKey` 只在 `LWH ⇄ WLH` 之间交替，**这是物理必然**（底面轮廓只有 L×W 与 W×L 两种）。区分四个状态的是 `orientationAxes`，**不是** `orientationKey`。任何"用 orientationKey 判断转了几下"的逻辑都是错的。

### 1.2 Shift+R = 上下旋转（`rotateBoxDown90`），绕世界 `x` 轴（柜长水平轴）

| 步骤 | orientationAxes |
|---|---|
| 基准 | `L+ W+ H+` |
| S ×1 | `L+ H+ W-` |
| S ×2 | `L+ W- H-` |
| S ×3 | `L+ H- W+` |
| S ×4 | `L+ W+ H+` |

## 2. 标签朝向需求（验收口径）

用户在**三个固定正交视角**下验收：俯视 top（从 z+ 向下看）、正视 front（沿 y 看 x-z 面）、侧视 side（沿 x 看 y-z 面）。
"正对相机的那个面"上的 A，其相对屏幕的旋转角度记为该面的 **label 角度**，取值 `0 | 90 | 180 | 270`（顺时针为正，与现有 2D `transform="rotate(...)"` 一致）。

### 2.1 R（左右旋转）

- **俯视看到的顶面**：A 角度随 R 走 `0 → 270 → 180 → 90 → 0`（四步**全不同**）。
- **四个竖直侧面**：在各自正对视角下 A **恒为正立（0°）**。
  → 这就是用户说的"前后左右四面的 A 始终一样"。

### 2.2 Shift+R（上下旋转）

- **侧视看到的侧面**：A 角度随 Shift+R 走 `0 → 270 → 180 → 90 → 0`（四步全不同）。
- 与翻转轴垂直、未被翻转影响的那两个面：A 恒正立。

### 2.3 组合（先 Shift+R 再 R）—— 本次最关键

组合后任意视角的 A 角度，**必须由统一函数 `faceLabelRotation(orientationAxes, view)` 唯一决定**。
当前 2D 走该函数、3D 走 quaternion 贴图，两套机制在组合旋转下不一致——这是用户报告"先 Shift+R 再 R 更不正确"的根因，必须消除。

## 3. 必须满足的契约

### 契约 C1 —— 唯一真值源 `faceLabelRotation`

- 2D（`ManualPlacement2D`、`ContainerPlan2D`）与 3D（`ContainerScene`）的标签朝向**必须出自同一函数** `faceLabelRotation(orientationAxes, view)`，签名与返回值域不变（`'top'|'front'|'side'` → `0|90|180|270`）。
- 不允许任何一侧再走独立的角度逻辑（例如 3D 单独用 `box.labelRotationDeg` 或单独算贴图角度）。
- 该函数**纯函数**，仅依赖 `orientationAxes` 与 `view`，不依赖 `orientationKey`、`labelRotationDeg`、`yawQuarterTurn`、`pitchQuarterTurn`。

### 契约 C2 —— `front` 视图不得硬编码为 0（当前 bug）

`orientationTransform.ts:94` 现状 `if (view === 'front') return 0` 是错误的。`front` 必须像 `top`/`side` 一样从 `orientationAxes` 推导。其物理定义见 §4。

### 契约 C3 —— 3D 标签必须靠真实旋转，不得用"上下左右一律相同的贴图"掩盖

3D 已用 quaternion 整体旋转箱体（`boxOrientationQuaternion`）。要求：**贴在每个面的 A 贴图，其在该面局部坐标系中的朝向，叠加箱体 quaternion 之后，在三个验收视角下的角度必须等于 `faceLabelRotation` 的返回值**。
实现上至少有两种合法做法，任选其一但必须通过 §5 的 3D 断言：
1. 仍贴"正立 A"到几何体本地各面，但**令几何体本地各面与本体轴 L/W/H 严格绑定**（即贴图朝向跟随本体面，而非世界面），由 quaternion 自然带到正确角度；
2. 或按面计算贴图旋转角，使其等于 `faceLabelRotation`。
无论哪种，§5 断言为准。

## 4. `faceLabelRotation` 的精确定义（实现必须复现下表，逐格为测试）

定义"视角的屏幕右方向 / 屏幕上方向"对应的世界轴：

| view | 屏幕右 = 世界 | 屏幕上 = 世界 | 正对的面法线 = 世界 |
|---|---|---|---|
| top   | x+ | y+ | z+ |
| front | x+ | z+ | y-（朝相机） |
| side  | y+ | z+ | x+（朝相机） |

A 贴纸的"正立"定义：贴纸的**上边缘指向本体 `H+`，右边缘指向本体 `L+`**（当某面不含某本体轴时，用该面所含的两条本体轴按右手序推导；以下表为唯一裁决）。

`faceLabelRotation` 返回**把正立 A 旋转到当前姿态所需的屏幕顺时针角度**。

> ⚠️ 重要：本规格**不提供手写的全量角度表**。原因：组合旋转下的角度必须由几何严格推导，任何手敲的数字都可能错（编写本规格时已发现初稿的组合表存在错误格）。因此真值由下方 §4.2 的**几何 oracle 算法**定义，测试用 `faceLabelRotation` 去比对 oracle，而不是比对人工填写的数字。

### 4.1 已知正确样例（来自现有通过测试，必须保留）

这两组是当前已通过的单测，视为"金标准锚点"，新实现必须继续满足：

- R 序列的 **top**：基准/R×1/R×2/R×3 = `0 / 270 / 180 / 90`
- Shift+R 序列的 **side**：S×1/S×2/S×3/S×4 = `270 / 180 / 90 / 0`

（其余视角的值由 §4.2 oracle 给出，不在此手写。）

### 4.2 几何 oracle（真值定义；测试中以独立实现复现，与 `faceLabelRotation` 对拍）

oracle 必须**独立于** `faceLabelRotation` 实现（放在测试文件内，或单独 `*.oracle.ts`），用基础线性代数算出角度，步骤：

1. **箱体旋转矩阵** `Rb`：由 `orientationAxes` 构造——世界轴 `x/y/z` 各列填入对应本体轴单位向量（带符号）。本体基向量 `L=(1,0,0) W=(0,1,0) H=(0,0,1)`。
   例：`orientationAxes={x:'W+',y:'L-',z:'H+'}` ⟹ `Rb` 列 = `[W, -L, H] = [(0,1,0),(-1,0,0),(0,0,1)]`。
2. **正立贴纸的本地朝向**：定义贴纸"上"= 本体 `H`，贴纸"右"= 本体 `L`（二者所在面之外的处理见第 4 步裁决）。旋转后世界向量 `up_w = Rb·H`，`right_w = Rb·L`。
3. **取当前正对相机那个面**，把 `up_w`、`right_w` **投影到该视角的屏幕平面**（用 §4 第一张表的"屏幕右/屏幕上 = 世界 ?"基向量做点积）：
   - `screenUp = ( up_w·右轴 , up_w·上轴 )`
   - 角度 = `atan2(screenUp.x, screenUp.y)` 归一到 `{0,90,180,270}`（顺时针为正）。
4. **裁决哪条本体轴当"右"/"上"**：当贴纸所在面不含 `H` 或 `L` 时（例如顶面只含 L、W），按该面所含两条本体轴、以"面法线朝相机、右手系"重新取 up/right。**此裁决规则必须在 oracle 注释中写死并举一例**；§4.1 的金标准锚点用于校准该规则方向是否取反。

> codex 实现 `faceLabelRotation` 后，测试对**所有 §1.1 / §1.2 序列 + §4.3 组合序列**的每个 (axes, view) 调用，断言 `faceLabelRotation(axes,view) === oracle(axes,view)`，并额外断言 §4.1 锚点的具体数字（防止 oracle 与实现一起整体转向错误）。

### 4.3 必须覆盖的组合序列（防回归核心，值由 oracle 给出）

先 `Shift+R ×1` 得 `L+ H+ W-`，在此基础上连续 R 三次，得到：
`H+ L- W-` → `L- H- W-` → `H- L+ W-`。
对这三个姿态 × `top/front/side` 共 9 个组合，全部以 oracle 为准逐一断言。**这是用户报告"先 Shift+R 再 R 更不正确"的精确复现点**，必须有断言。
